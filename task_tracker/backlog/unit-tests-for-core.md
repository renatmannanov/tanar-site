# Unit-тесты на репозитории `src/core/*`

> Тип: инфраструктура тестов
> Статус: backlog
> Дата: 2026-05-23

## Контекст

В Фазе 0 (фундамент данных) сознательно ограничились существующими e2e (Playwright). E2e покрывают поведение сайта, но не ловят регрессии в логике конкретных запросов к БД (Drizzle-репозитории `core/catalog`, `core/inventory`, `core/orders`).

Когда в репозиториях появится сложная логика (резервирование остатков, отмена с возвратом резерва, фильтры по наличию, мульти-источниковые заказы) — e2e уже не вытянут, нужны прицельные unit-тесты.

## Что сделать

1. Подключить **vitest** (выбран как родной для Vite-стека и Next.js 15, проще jest).
2. Тестовая БД через `postgres-test` из `docker-compose.yml` (уже заведена в Фазе 0, порт 5433).
3. Хелпер `setupTestDb()`:
   - применяет миграции drizzle-kit к чистой test-БД,
   - перед каждым тестом — TRUNCATE всех таблиц (быстрее чем re-migrate),
   - seed-функции для типовых сценариев (товар с вариантом, товар с остатками, заказ в разных статусах).
4. Покрыть unit-тестами:
   - `core/catalog` — getProductBySlug, getProductsByCategory, фильтры
   - `core/inventory` — резерв/списание/отмена, edge-cases (двойной резерв, отмена несуществующего)
   - `core/orders` — создание заказа со списанием остатков (транзакция)
5. CI-step: `npm run test:unit` (отдельно от `test:e2e`), запускать на dev-ветке.

## Когда брать

После Фазы 2 (остатки) и до Фазы 3 (корзина) — потому что логика корзины полагается на корректные резервы, без unit-тестов на inventory это хрупко. Можно и раньше, если в Фазе 1/2 логика репозиториев усложнится.

## Не делать сейчас (в Фазе 0)

В Фазе 0 ограничиваемся e2e — фундамент простой (CRUD-репозиторий + миграция данных), e2e ловит регрессии достаточно.
