## Порядок поставки (зафиксировано 2026-05-29)

Каталог + админка разбиты на ТРИ плана. Фундамент проектируется ОДИН раз под полный объём, UI поставляется по частям — без переделки.

- **План A — Импорт + фундамент CRUD** (этот документ): расширение схемы, чистка демо, снапшот CSV, **полный write-контракт `core/catalog`** (create/update/delete + вложенные variant/sku, транзакции, zod-валидация), импорт-скрипт наполняет БД ЧЕРЕЗ эти CRUD-методы (первый потребитель → контракт обкатан боевыми данными). Заготовка контракта `core/media`.
- **План B — Админка: редактирование** (Фаза 1, этап 2): shell + auth + реестр разделов; универсальная форма товара `mode: create|edit`, открыта на edit; кнопки create/delete и слот фото видны, но `disabled` (ловим визуальные косяки раскладки сразу).
- **План C — Полный CRUD + фото** (этап 3): снимаем `disabled` (формы/actions готовы), оживляем media-picker + sharp-загрузку.

> Гарантия «без переделки»: редактирование (B) — подмножество CRUD, чьи контракты (A) и формы (B) написаны сразу в полном виде; этап 3 только включает уже существующее.

## Решения (зафиксировано 2026-05-29)

### 1. Маппинг категорий витрины — РЕШЕНО
Демо-категории (`jackets|hoodies|t-shirts|pants|shorts`) **выбрасываются**. Категории заводятся **по факту из CSV** — структура заказчицы как есть:
**куртки / брюки / шорты / футболки / поло** (русские ярлыки; коды-слаги категорий уточнить по реальному CSV).
- `hoodies` в каталоге нет → удаляется из categories (не оставляем пустым).
- Поло → отдельная категория `polo` (по факту CSV).
- Заказчица «худи положила в куртки» — категория определяется её разбивкой в CSV, не нашим домыслом.
> ⚠️ Точный список категорий и распределение моделей **сверить по реальному CSV** в начале плана A (CSV не в контексте — gitignored).

### 2. Расширение схемы — РЕШЕНО
**В core/catalog (грузим сейчас):**
- `skus.article` (TANAR-001) — внутренний артикул на уровне SKU.
- `skus.ru_size` (46/48/50) — рос. размер параллельно буквенному (`skus.size`).
- `products.label` — короткий tech-лейбл с бирки (GORE-TEX®, WINDBREAKER, UPF 30+).
- `products.care` — уход за изделием (отдельный текст).

**НЕ грузим сейчас → Фаза 5 (`MarketplaceListing`):**
- `SKU Ozon` (4373879096) — внешний ID площадки.
- `ID группы Ozon` — внешний group id (используется при импорте ТОЛЬКО для группировки строк в товар, в БД не пишется).
- `Цена Ozon (факт)` — цена канала.

**Страховка от потери исходника:** копия CSV **коммитится в репо как снапшот-источник** (не gitignored). Фаза 5 маппит Ozon-поля из снапшота → `MarketplaceListing`, повторным прогоном импорта. Исходник не теряется.
> ⚠️ Снапшот содержит цены/артикулы и попадёт в git-историю. Репо приватное — приемлемо. Альтернатива (отдельный приватный data-репо) — решить при написании плана A.

### 3. Остатки — РЕШЕНО
Импортируем на уровень **SKU как в CSV** (цвет×размер, число в число). По данным заказчицы в таблице остаток посаймовый («Чёрный M — 19, Чёрный L — 19»), а не на варианте — разбор выше (старая версия «на варианте») был неточен.
Мусорные значения («—»/«уточнить», ~10 SKU) → `stockQty = 0`.
> ⚠️ Разбивку остатков (SKU vs вариант) **сверить по реальному CSV** в начале плана A — истина в самом файле.

### 4. Slug-генерация — РЕШЕНО
**Ручная** схема на 12 товаров (`jacket-sv7-goretex`, `pants-2006-winter`, …). Товаров мало — составляем читаемые слаги вручную при написании плана A (после чтения CSV).

### 5. Цены — РЕШЕНО
Базовая цена витрины Tanar = **Цена Kaspi** → `products.priceBase`. Ozon-цена — цена канала, в core не грузится (Фаза 5).

### 6. Фото
Реальных товарных фото пока НЕТ (есть 3 сгенерированных, «так себе»). **Файлы в `public/images/` при чистке демо НЕ трогаем.** До появления фото — CSS-градиенты/placeholder по конвенции путей. Реальные фото — заказчица зальёт через админку (план C, sharp-пайплайн).
