# Архитектура: tanar-site → самостоятельная e-commerce площадка

> Тип: архитектурный фундамент (не план — основа для будущих планов)
> Статус: проектирование
> Дата: 2026-05-23
> Решение: Вариант А (сайт — источник правды)

## Зафиксированные решения (вход)

| Вопрос | Решение |
|---|---|
| Модель бизнеса | **Вариант А** — сайт самостоятельная площадка, сам ведёт каталог + остатки |
| Хостинг | **VPS** (свой сервер) — можно долгоживущие воркеры, любая БД |
| Хранилище данных | **Своя БД** (Postgres) — уход от `src/data/products.ts` |
| Размеры | Есть (S/M/L и т.п.), остатки на уровне **размер × цвет** |
| Учёт остатков | Продали → списалось автоматически; ручная корректировка в админке; в будущем — интеграция с CRM |
| Порядок построения | **1) Админка (каталог + остатки) → 2) Корзина → 3) Оплата** |
| Маркетплейсы | Будущий **канал раздачи**, не делаем сейчас, но архитектуру не закрываем |
| Онлайн-оплата | Будущее, заложить место, не реализовывать |
| **Структура** | **Modular monolith** — один деплой, строгая модульность на уровне кода/папок. **Старт: логическая модульность внутри `src/`** (границы через структуру + ESLint), **БЕЗ** pnpm workspaces и отдельных `package.json`. Переход на настоящую монорепу — когда появится второй проект (это механическая операция, не переделка) |
| **БД / ORM** | **Postgres + Drizzle**. Близость к SQL, схема на TypeScript, лёгкие миграции. Без Prisma (тяжёлый, своя «магия»). Без raw SQL (теряем DX) |
| **МП-модули** | 3 отдельных модуля (Kaspi/Ozon/WB), реализуют общий интерфейс, **вкл/выкл через env/конфиг** на деплое |
| **Контент сайта** (картинки везде + блог-редактор) | В **бэклог** — отдельные планы поверх админки (см. [admin-content-management.md](./admin-content-management.md)) |

> **Важно:** это отменяет правило старого CLAUDE.md «никакого бэкенда, никакой БД». Оно было верным для витрины v1, но целевой Вариант А требует БД. CLAUDE.md обновить при старте реализации.

## Главный вывод из research по API маркетплейсов

Все три МП работают по модели **«сайт = источник правды, мы пушим/отдаём данные им»** — это идеально под Вариант А. МП = выходные каналы, не конкурирующие источники.

| | Kaspi | Ozon | Wildberries |
|---|---|---|---|
| Создать товар | REST / XML-фид | `/v3/product/import` | `/content/v2/cards/upload` |
| Остатки | **pull** (Kaspi тянет наш XML ~60мин) | **push** REST | **push** REST |
| Цены | XML-фид | push | push |
| Заказы | polling REST | FBS/FBO | FBS |
| Sandbox | ❌ нет | ✅ | ✅ |
| Боль | нет REST для остатков | маппинг категорий | маппинг + модерация 2-24ч + Честный Знак |

**Следствия для архитектуры (заложить сейчас, реализовать потом):**
1. Синхронизация — **две модели**: Kaspi через генерируемый XML-фид по публичному URL (просто), Ozon/WB через фоновый push-воркер.
2. У каждого МП своя таксономия + обязательные атрибуты → в модели товара нужно место под **per-marketplace маппинг** (категория, атрибуты, артикул/SKU на каждой площадке, флаг «выгружать сюда»).
3. Заказы могут приходить из 4 источников (сайт + 3 МП) → модель `Order` должна иметь поле `source`.

## Модульная структура (modular monolith)

Один деплой, но код разрезан на **изолированные модули** с явными контрактами. Главный принцип: **модуль общается с другими ТОЛЬКО через публичный контракт (его `index.ts` / интерфейсы), не лезет во внутренности.** Тогда модуль можно вынести/скопировать в другой проект.

### Раскладка СТАРТА (логическая модульность внутри `src/`)

Один `package.json`, один Next.js проект, но дисциплина границ через структуру папок и ESLint. Целевая раскладка такая, чтобы потом папки можно было механически перенести в `packages/` без переделки кода:

```
src/
  core/                  ← ДОМЕННОЕ ЯДРО, ни от кого не зависит
    catalog/             каталог: Product/Variant/Sku, чтение/запись
      index.ts           публичный API модуля (только через него импортируют другие)
    inventory/           остатки: stockQty, reservedQty, резерв/списание, InventoryLog
      index.ts
    orders/              заказы: Order/OrderItem, статусы, источник (source)
      index.ts
    media/               фото: загрузка, sharp-пайплайн, хранение
      index.ts
    db/                  Drizzle: схема + миграции + клиент (Postgres)
      schema.ts          таблицы (products, variants, skus, orders, ...)
      client.ts          инстанс drizzle-клиента
      migrations/        SQL-миграции (drizzle-kit)
      index.ts
    contracts/           ОБЩИЕ ИНТЕРФЕЙСЫ: типы, события, порты (см. ниже)
      index.ts

  marketplace/           ← МП-МОДУЛИ (Фаза 5, места под них заводим сразу)
    contract/            MarketplaceModule интерфейс (общий для всех трёх)
    kaspi/               (Фаза 5) XML-фид + polling заказов
    ozon/                (Фаза 5) push API
    wildberries/         (Фаза 5) push API
    registry.ts          собирает включённые модули из env (MARKETPLACES=...)

  app/                   ← Next.js App Router (УЖЕ ЕСТЬ)
    (public)/            витрина: каталог, товар, корзина
    (admin)/             админка: CRUD, остатки, фото, заказы (Фаза 1+)
    api/                 route handlers (заказы, медиа-аплоад, и т.п.)

  components/            ← UI-компоненты (УЖЕ ЕСТЬ, разнесём по public/admin)
  lib/                   ← общие утилиты (НЕ доменная логика)
```

> **`web` и `admin` — сегменты одного Next.js приложения** через route groups `app/(public)/` и `app/(admin)/`. Авторизация админки — middleware на `(admin)`. Разнесение на разные деплои — потом.

### Дисциплина границ (как держим модули изолированными без монорепы)

1. **Каждый модуль имеет `index.ts`** — это его публичный API. Всё, что НЕ экспортировано из `index.ts`, считается внутренним.
2. **ESLint-правило** `no-restricted-imports` запрещает импортировать внутренности модуля минуя его `index.ts`. Например, нельзя `import x from '@/core/catalog/internals/foo'` — только `import { ... } from '@/core/catalog'`. (Аналогично уже сделано для `@/data/products` в текущем рефакторе.)
3. **Направление зависимостей контролируется ESLint**: `app/*` → `core/*` и `marketplace/*`; `marketplace/*` → `core/contracts`; `core/*` ни от кого не зависит снаружи. Цикл/обратная зависимость = ошибка линтера.
4. **При будущем переезде в монорепу** каждый модуль = отдельный `packages/<name>/`, `index.ts` уже на месте, импорты `@/core/catalog` меняются на `@tanar/core-catalog` через скрипт. **Кода трогать не надо.**

### Контракты между модулями (ключ к переносимости)

- `core/contracts` определяет **порты** (интерфейсы), а реализации внедряются (dependency injection / простой реестр). Пример: `OrderSink`, `InventoryReader`, `MediaStore`.
- **МП-модуль реализует единый интерфейс** `MarketplaceModule`:
  ```ts
  interface MarketplaceModule {
    id: 'kaspi' | 'ozon' | 'wildberries';
    isEnabled(): boolean;                       // читает env/конфиг
    pushCatalog(products: Product[]): Promise<SyncResult>;
    pushStock(skus: Sku[]): Promise<SyncResult>;
    pullOrders(): Promise<Order[]>;             // или отдаёт фид (Kaspi)
    updateOrderStatus(o: Order, s: Status): Promise<void>;
  }
  ```
  Ядро не знает про Kaspi/Ozon/WB конкретно — только про массив `MarketplaceModule[]`, собранный из включённых по env.
- **Вкл/выкл:** реестр модулей читает env (`MARKETPLACES=kaspi,ozon`) → подключает только их. Выключенный МП-модуль не грузится, его отсутствие ничего не ломает.
- **Направление зависимостей:** `apps/*` → `core/*` и `marketplace/*`; `marketplace/*` → `core/contracts` (но НЕ наоборот — ядро не зависит от МП). Витрина НЕ зависит от админки и наоборот.

### Поток данных

```
apps/web (витрина) ─┐
apps/admin ─────────┤→  core (catalog/inventory/orders/media)  →  db (Postgres)
                    │                    │
                    │                    └→ media-файлы на VPS
                    ▼
            marketplace registry (env-driven)
                    ├─ kaspi  → XML-фид по URL (Kaspi тянет) + polling заказов

```

## Архитектура админки (учитывает контент-менеджмент СРАЗУ)

Админка проектируется не как «панель для каталога», а как **оболочка для управления любым ресурсом сайта**. Контент-менеджмент (картинки сайта + блог) реализуется в Фазе 6, но админка с Фазы 1 строится так, чтобы он встроился без переделки.

**Принцип:** админка = набор **resource-разделов** под общим каркасом (layout, auth, навигация, формы, загрузчик медиа). Каждый раздел — каталог, остатки, заказы, **медиа сайта, блог** — реализует общий контракт `AdminResource`, а не хардкодится.

```
src/app/(admin)/
  layout.tsx             ← admin layout + auth-guard (middleware на (admin))
  shell/                 ← каркас: layout, навигация (реестр разделов), общие формы, media-picker
  catalog/               ← Фаза 1
  inventory/             ← Фаза 2
  orders/                ← Фаза 3
  site-media/            ← Фаза 6 (картинки hero/story/лого) — место заложено сейчас
  blog/                  ← Фаза 6 (редактор постов) — место заложено сейчас
  marketplaces/          ← Фаза 5 (статус/настройки синка) — место заложено сейчас
```

**Что закладывается с Фазы 1 (чтобы контент влез потом):**
- Навигация админки — **реестр разделов**, добавление раздела = регистрация, не правка меню вручную.
- **Media-picker / загрузчик** (`core/media`) — общий компонент: используется и каталогом (фото товара), и позже site-media (hero/story), и блогом (картинки в посте). Один загрузчик → sharp-пайплайн → хранилище. Не делать отдельный загрузчик под каждый раздел.
- **`core/media` управляет ВСЕМИ изображениями сайта**, не только товарными. Модель `MediaAsset` (см. схему) с полем `scope` (product | site | blog) — заведена сразу, наполняется по фазам.
- Блог: контент остаётся MDX/структурой, но админка-раздел `blog` подключится к тому же media-picker и тому же auth/layout.

> Итог: в Фазе 1 пишем `shell` + `resources/catalog`, но `shell` уже умеет реестр разделов и media-picker. Фаза 6 = добавить `resources/site-media` и `resources/blog`, ядро не трогаем.

## Модель данных (черновик схемы)

Ключевая идея: товар → варианты (цвет) → SKU (цвет × размер) — остаток живёт на **SKU**.

```
Product
  id, slug, name, category, description, specs(jsonb), priceBase, currency,
  status (draft|published|archived|coming_soon),
  label?,                 // [real-catalog-import] tech-лейбл с бирки (GORE-TEX®, UPF 30+)
  care?,                  // [real-catalog-import] уход за изделием
  createdAt, updatedAt

ProductVariant            // цвет
  id, productId, colorId, colorLabel, hex, models[], hasFlatShots

Sku                       // цвет × размер — единица учёта остатков
  id, variantId, size,
  article?,               // [real-catalog-import] внутренний артикул (TANAR-001)
  ruSize?,                // [real-catalog-import] рос. размер (46/48) рядом с буквенным
  priceOverride?, barcode?,
  stockQty,               // фактический остаток
  reservedQty             // зарезервировано неоформленными/оформленными заказами

MediaAsset                // ВСЕ изображения сайта (товары, hero/story/лого, блог)
  id, scope(product|site|blog), url, sortOrder,
  // продуктовые поля (когда scope=product):
  productId?, variantId?, view?, model?, role(lifestyle|flat)?,
  // site/blog: смысловой ключ (напр. 'home.hero', 'story.1', 'blog:post-slug')
  key?, alt?

Order
  id, source(site|kaspi|ozon|wb), status, contact, customer(jsonb),
  total, createdAt
OrderItem
  id, orderId, skuId, nameSnapshot, priceSnapshot, qty

InventoryLog              // аудит движения остатков (продажа, возврат, ручная правка)
  id, skuId, delta, reason, refOrderId?, createdAt

// БУДУЩЕЕ (заложить, не делать):
MarketplaceListing
  id, productId|skuId, marketplace(kaspi|ozon|wb),
  externalId, externalSku, categoryMap(jsonb), attributesMap(jsonb),
  enabled, lastSyncedAt, syncStatus
```

### Как «наличие» считается и показывается (ответ на исходный вопрос)
- Источник правды — `Sku.stockQty`.
- Доступно к покупке = `stockQty - reservedQty > 0`.
- На витрине: размер показывается доступным/недоступным; вариант (цвет) недоступен если все его размеры = 0.
- При оформлении заказа: `reservedQty += qty`. При подтверждении/отгрузке: `stockQty -= qty`, резерв снимается. При отмене: резерв снимается без списания.
- Все движения — в `InventoryLog` (откуда что списалось).
- Ручная правка остатка в админке = запись в `InventoryLog` с reason=manual.

## Закрытые решения (зафиксированы по ходу обсуждения)

- ✅ **БД:** Postgres.
- ✅ **ORM:** Drizzle.
- ✅ **Структура:** логическая модульность внутри `src/` (без pnpm workspaces сейчас).
- ✅ **Фото:** хранить на VPS (файловая система) + текущий sharp-пайплайн. Загрузка через админку → прогон через sharp → public/. Связь: [assets-pipeline-extension](./assets-pipeline-extension.md).
- ✅ **Авторизация админки:** один пользователь (заказчица) → простой подход (NextAuth credentials или сессия + cookie). Без ролей. Конкретику выбрать в начале Фазы 1, не блокер для Фазы 0.
- ✅ **Миграция текущих данных:** one-off скрипт переносит все живые товары из `products.ts` + существующие фото в БД. Реализуется в Фазе 0.
- ✅ **Витрина во время перехода:** сайт остаётся живым. Витрина переключается на чтение из БД через `core/catalog`, поведение внешне не меняется.

## Решения по Фазе 1 (зафиксированы 2026-05-29)

- ✅ **Порядок:** импорт боевого каталога ПЕРЕД админкой. Импорт дешёвый (паттерн `seed.ts`), финализирует модель данных, не зависит от отсутствующих фото. Админка строится на боевых данных.
- ✅ **Объём админки:** полный CRUD заложен в фундамент (логика + контракты) СРАЗУ; UI поставляется поэтапно (A→B→C). Цель — не переделывать редактирование при добавлении создания/удаления/фото.
- ✅ **UI админки:** свой минимализм на **Tailwind v3 + лёгкие headless-примитивы** (Radix/shadcn точечно — формы/диалоги/комбобоксы). НЕ витринная outdoor-эстетика — функциональный внутренний инструмент. НЕ готовый admin-фреймворк (Refine/React-Admin навязали бы свой роутинг, конфликт с modular monolith и resource-каркасом).
- ✅ **Категории:** по факту из CSV (куртки/брюки/шорты/футболки/поло), демо-категории выбрасываются. См. [real-catalog-import.md](./real-catalog-import.md) §1.
- ✅ **Цена:** Kaspi = `priceBase`; Ozon-цена → Фаза 5.
- ✅ **Ozon-сырьё:** снапшот CSV в репо, маппинг → `MarketplaceListing` в Фазе 5 (не теряем исходник).
