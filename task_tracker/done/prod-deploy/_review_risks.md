## Критичное (только [CONFIRMED] и [LIKELY])

### 1. ALLOW_PROD_SEED повторный запуск — необратимая потеря данных [CONFIRMED]

**Файл:** step_6_seed_once.md, PLAN.md (архитектурные решения), progress.md (секция «Сид»)

seed.ts делает `TRUNCATE products, product_variants, skus, media_assets CASCADE` — это зафиксировано в progress.md строка ~31 и step_6 явно. Флаг `ALLOW_PROD_SEED=1` остаётся в .env прода (или в compose-сервисе), потому что его туда добавляют при первом запуске.

Риск: через неделю после первого сида кто-то (или агент) запускает `docker compose --profile tools run --rm seed` повторно — снапшот каталога не изменился, но команда `TRUNCATE ... CASCADE` уничтожает весь каталог **вместе с media_assets**, которые к тому моменту заполнены через push-media и/или ручную загрузку. Файлы webp в volume остаются осиротевшими, строки media_assets удалены.

**Что план говорит:** шаг 6 пишет «запускать ТОЛЬКО на чистой БД... После первого сида — НИКОГДА повторно». Но механизма, который физически предотвращает повторный запуск, нет. Флаг не удаляется автоматически. Предупреждение написано в md-файле; агент-исполнитель, выполняющий шаг 7 в будущем, его не читает.

**Откат:** невозможен без бэкапа. Бэкап (шаг 8) — отдельная фаза после деплоя; в момент потери данных он может не существовать.

### 2. drizzle-kit берёт DATABASE_URL из окружения без .env.local — не проверено [CONFIRMED]

**Файл:** step_5_migrations.md, progress.md (секция «БД / Drizzle»)

progress.md строка ~25: «`drizzle.config.ts` грузит `.env.local` через dotenv — на проде его нет». step_5 фиксирует это как открытый вопрос: «ПРОВЕРИТЬ, что drizzle-kit берёт DATABASE_URL из process.env при отсутствии .env.local».

Если `dotenv.config()` вызван с `{ path: '.env.local' }` (а не просто `.env`), и при отсутствии файла он молча не устанавливает переменную, то drizzle-kit получит пустой DATABASE_URL и либо упадёт с ошибкой соединения, либо попытается подключиться к localhost:5432 (нет такого внутри контейнера).

**Что план говорит:** «dotenv.config с несуществующим файлом не падает (просто не находит) — переменная из окружения подхватится». Это предположение, не факт из кода. Если dotenv вызывается с `{ override: true }` или аналогом, поведение может отличаться. Проверка помечена как «ПРОВЕРИТЬ» — значит, на момент написания плана это не было верифицировано.

**Последствие:** миграции не применятся на чистой прод-БД → web стартует, но все dynamic-страницы падают (таблиц нет) → 500 на `/catalog`, `/`.

### 3. Volume product-images: права на запись от USER node [CONFIRMED]

**Файл:** step_1_dockerfile.md, step_2_compose_prod.md, progress.md

step_1 (раздел runner): «чown -R node:node /app/public перед USER node, либо запускать от node и в шаге 2 задать владельца volume». step_2: «На первом up volume пустой и создаётся root'ом → возможен EACCES при загрузке фото». Решение «зафиксировано» — но в виде двух альтернатив, а не одного конкретного действия.

Риск: если в Dockerfile написать `chown node:node /app/public`, но volume монтируется ПОВЕРХ этой папки на старте контейнера, Docker создаёт директорию volume с правами root. `chown` в образе не распространяется на содержимое volume. Результат: `sharp` падает на `EACCES` при первой же загрузке фото через админку на проде.

**Что план говорит:** проблема признана, предложено два варианта. Ни один не зафиксирован как единственный. Агент-исполнитель должен будет выбрать — и может выбрать неправильно или пропустить этот пункт.

### 4. Порядок DNS → деплой: DNS не распространилось, Caddy пытается выпустить сертификат — ACME rate-limit [CONFIRMED]

**Файл:** step_7_first_deploy.md (шаг 7б), step_3_caddy.md, progress.md (секция «Caddy / SSL / DNS»)

progress.md строка ~42: «Порядок: DNS → деплой → первый https-запрос». step_7 пишет: «Дождаться распространения (`dig +short tanar.kz` с VPS показывает его IP)».

Риск: агент (или пользователь) запускает `docker compose up` до того как DNS полностью распространился (dig показывает нужный IP, но Let's Encrypt ACME-сервер с его IP всё ещё резолвит старый адрес). Caddy делает попытку выпустить сертификат → ACME-challenge HTTP-01 не проходит (запрос идёт на старый IP) → Caddy помечает попытку как failed → повторные попытки идут с backoff. Let's Encrypt rate limit: 5 неудачных попыток для одного домена за 1 час (failed validation rate limit). При частых перезапусках Caddy (попытках отладить) можно попасть в лимит и получить блокировку на 1 час.

Дополнительный риск: `.kz` домены управляются через КЦДИ (регистратор Kazakhstan). Некоторые `.kz` регистраторы имеют TTL по умолчанию 24 часа — `dig` с VPS может показать нужный IP через несколько часов, а не минут. Никакого ожидания кроме «dig подтверждает» в плане нет.

**Что план говорит:** предусловие зафиксировано правильно, но нет инструкции что делать если dig показывает нужный IP а сертификат всё равно не выпускается (как долго ждать, как проверить статус ACME через Caddy API).

### 5. Merge dev→main перед деплоем — предусловие без явного подтверждения [CONFIRMED]

**Файл:** step_7_first_deploy.md, progress.md (секция «Git»)

step_7в: `git checkout main` на сервере. progress.md: «Сейчас всё на dev, в main не влито. ПЕРЕД деплоем — отдельно влить dev→main по git-протоколу; план НЕ вливает автоматически». step_7 называет это «предусловием шага 7 (сделать осознанно)».

Риск: агент-исполнитель, выполняя шаг 7 в рамках ralph-loop, видит `git checkout main` и выполняет его. На сервере main — пустая ветка (или старый код). Деплой разворачивает устаревший код без всего того, что собиралось в шагах 1–6 на dev. При этом `docker build` успешно соберёт образ из старого кода — ошибки не будет; проблема обнаружится только при функциональной проверке.

**Что план говорит:** предусловие описано, но в чеклисте шага 7 оно стоит ПОСЛЕДНИМ пунктом («merge dev→main выполнен осознанно ПЕРЕД деплоем»), а не первым. Порядок выполнения ralph-loop — по тексту шага, а команды `git checkout main` стоят в разделе 7в (до проверок).

## Важное (только [CONFIRMED] и [LIKELY])

### 6. Имя volume в backup.sh: `tanar-site_product-images` vs реальное имя [LIKELY]

**Файл:** step_8_backups.md

step_8: «имя volume — проверить `docker volume ls`, обычно `<project>_product-images`». Имя проекта Docker определяется именем директории на сервере или `COMPOSE_PROJECT_NAME`. Если пользователь клонировал репо не в `tanar-site/`, а в `tanar/` или `app/`, имя volume будет другим.

Риск: `backup.sh` с захардкоженным именем volume `tanar-site_product-images` будет работать только пока репо лежит в директории `tanar-site`. При переносе, переименовании, или новом деплое — `docker run ... -v tanar-site_product-images:/data` молча создаст новый пустой volume (Docker не выдаёт ошибку при несуществующем volume — просто создаёт), бэкап фото будет пустым `tar.gz`.

**Что план говорит:** «проверить `docker volume ls`» — но это ручная инструкция, не автоматическая проверка в скрипте.

### 7. Доступ к прод-БД из push-media CLI: механизм не зафиксирован окончательно [CONFIRMED]

**Файл:** step_9_push_media_cli.md

step_9 описывает два варианта и в конце фиксирует архитектуру: «push-media = две части: (1) локально генерит webp+манифест, (2) на проде сервис `push-media` (profile tools) читает манифест и пишет в БД». Но формат манифеста, способ передачи манифеста на VPS и сервис `push-media` в compose — нигде не описаны как готовые артефакты. Шаг 9 говорит «Зафиксированный дизайн» для некоторых частей, но сам механизм передачи манифеста (scp? rsync? docker cp json-файла?) остаётся открытым.

Риск: агент-исполнитель шага 9 будет принимать архитектурные решения на месте без явных требований → возможен несогласованный результат или потраченное время на согласование с пользователем в процессе.

### 8. Turbopack + standalone: артефакт .next/standalone может не создаться [CONFIRMED]

**Файл:** progress.md, step_1_dockerfile.md

progress.md: «`next build --turbopack` (есть в package.json `build`). Turbopack + standalone — проверить, что `.next/standalone` создаётся (на старых версиях standalone был только webpack; Next 15.5 — поддерживает, но ПРОВЕРИТЬ артефакт после build)».

step_1: «Проверить, что создаётся `.next/standalone`».

Риск: если Turbopack в текущей версии Next 15 не создаёт `.next/standalone` (или создаёт неполный артефакт), стадия `builder` в Dockerfile скопирует несуществующую директорию → `COPY --from=builder /app/.next/standalone ./` не упадёт на build (COPY несуществующего в многостадийной сборке — Docker поведение зависит от версии), но runner-контейнер стартует без `server.js` → `node server.js` падает сразу.

**Что план говорит:** предупреждение есть, проверка указана как критерий готовности шага 1. Риск управляемый при выполнении, но если агент пропустит — деплой упадёт.

### 9. POSTGRES_PASSWORD дублируется в DATABASE_URL — рассинхрон [CONFIRMED]

**Файл:** step_4_env_secrets.md

step_4: «`POSTGRES_PASSWORD` в `DATABASE_URL` и в секции postgres ДОЛЖНЫ совпадать. Документировать это (или вынести в один источник — но compose не подставляет env в env, поэтому дублируем явно и предупреждаем)».

Риск: пользователь при заполнении `.env` меняет `POSTGRES_PASSWORD`, но забывает обновить его же внутри строки `DATABASE_URL`. web-контейнер получит ошибку аутентификации к postgres («password authentication failed»). Контейнер стартует (depends_on ждёт healthcheck postgres, а postgres стартует успешно), но все запросы к БД падают → 500 на всех dynamic-страницах.

### 10. Бэкап cron: директория backups/ не создана заранее [LIKELY]

**Файл:** step_8_backups.md

`backup.sh` пишет в `backups/db-$TS.sql.gz`. Если директория `backups/` не существует на VPS (она не создаётся ни в одном из шагов 1–7, и не входит в репо через gitignore), первый запуск cron-задачи завершится ошибкой `No such file or directory`. Ошибка пишется в `backups/backup.log` — который тоже не создастся.

**Что план говорит:** `mkdir -p backups` нигде не упомянут ни в шаге 8, ни в шаге 7.

### 11. ufw enable без проверки SSH-правила — потенциальная блокировка [LIKELY]

**Файл:** step_7_first_deploy.md (шаг 7а)

```bash
sudo ufw allow 22 && sudo ufw allow 80 && sudo ufw allow 443 && sudo ufw enable
```

Риск: если сервер слушает SSH не на порту 22 (некоторые провайдеры, в т.ч. ps.kz, меняют стандартный порт), то `ufw enable` заблокирует текущий SSH-сеанс. Управление сервером потеряно (только через VNC/KVM консоль провайдера).

**Что план говорит:** порт 22 указан явно без проверки реального SSH-порта. Для PS.kz это предположение.

## Мелочи ([THEORETICAL])

### T1. Let's Encrypt rate limit при staging-тестировании

Если при отладке деплоя стек несколько раз поднимался/опускался с `{$DOMAIN}` (без staging Caddy), и каждый раз Caddy пытался выпустить сертификат — можно попасть в лимит 5 cert/domain/week. В плане нет инструкции использовать Caddy staging CA для тестирования. [THEORETICAL] — зависит от поведения агента при отладке.

### T2. mdx-файлы блога в standalone: content/blog не скопирован

Standalone-артефакт не включает файловую систему вне `.next/`. Blog MDX (`content/blog/*.mdx`) читаются `fs` в рантайме. Если `content/` не скопирован в runner-стадию — `/blog` упадёт. Не найдено явного указания в плане. [THEORETICAL] — зависит от того, как реализован рантайм-чтение blog MDX (может быть уже встроен в standalone статикой).

### T3. Размер образа и время сборки на VPS (2 vCPU / 4 ГБ)

`docker build` на сервере с Next.js + Turbopack + node:20-slim может занять 5–15 минут. При compile-ошибке придётся пересобирать. Не блокер, но не предупреждено. [THEORETICAL]
