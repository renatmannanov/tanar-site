## Важное (исполнитель потеряет >5 минут на разбирательство)

### 1. Шаг 4 зависит от шага 3 по «содержанию файла», а не только по «порядку запуска» — но реальная зависимость шире

**Где:** `step_4_storefront_links.md` → `> Зависит от: шаг 3 (e2e создаёт ссылки через таб админки; тот же спек-файл)`

**Что не так:** зависимость указана только потому, что «тот же спек-файл» и «e2e создаёт данные». Но шаг 4 также меняет `ProductDetail.tsx` — а шаг 3 меняет `ProductForm.tsx`. Это разные файлы, конфликта нет. Однако e2e-тесты шага 4 используют данные, которые шаг 3 записывает в БД через сохранение формы (тест «Сохранение» в блоке шага 3). Если агент прогоняет шаг 4 без полного завершения шага 3 (например, UI сделан, но тест сохранения пропущен), витринный e2e упадёт с пустыми ссылками. Зависимость в поле указана правильно, но причина (данные в БД, а не только файл) не объяснена — агент может решить, что шаги параллельны после «я уже добавил UI таба».

**Потеря:** агент запускает витринный e2e без записанных SKU-ссылок → тест падает → 5-10 минут дебага «почему href не подменяется».

**Исправление:** в `step_4`, поле «Зависит от» дополнить: `шаг 3 полностью (включая e2e-тест сохранения — витринные тесты нужны данные в БД)`.

### 2. Шаг 2: условие «109 записей» упадёт на dev-БД после `db:seed` без `db:seed-mp-links`

**Где:** `step_2_links_import.md`, команды верификации:
```
docker exec ... "SELECT count(*) FROM skus WHERE marketplaces ? 'ozon' ..."   # 109
```
и критерий: `Сидер на dev: updated 109, идемпотентен (повторный запуск без изменений)`.

Но в `progress.md` явно написано: `admin.spec и admin-marketplaces.spec в afterAll гоняют npm run db:seed → ВСЕ таблицы перезаливаются (включая sku.marketplaces)`.

**Что не так:** если агент запустит `npm run test:e2e` после прохождения шага 2 (что требует шаг 2 для `typecheck/lint/build`), afterAll в admin-спеках сбросит `sku.marketplaces → {}` для всех строк. Критерий готовности шага 2 `SQL: 109 SKU с обеими ссылками` стало невалидным, но агент не знает что надо перегнать сидер после прогона всего e2e.

Примечание в `progress.md` говорит «перезапустить `db:seed-mp-links`, если проверяешь руками» — но это спрятано в `progress.md` и не входит в команды верификации шага 2.

**Потеря:** агент прогоняет `test:e2e` → SQL-проверка выдаёт 0 → агент думает сидер сломался → 5-15 минут дебага.

**Исправление:** добавить в команды верификации шага 2 пояснение: `# после test:e2e перезапустить: npm run db:seed-mp-links`.

### 3. Шаг 1: поле `Sku.marketplaces` в types.ts — тип `optional` vs обязательный

**Где:** `step_1_schema_contract.md`, раздел «2. Read-контракт»:
> `Sku`: добавить поле `marketplaces?: Partial<Record<Marketplace, string>>`

Но схема БД: `NOT NULL DEFAULT '{}'` — значит поле всегда присутствует (никогда не NULL).

**Что не так:** если тип объявлен как `?` (optional/undefined), то в шаге 4 выражение `selectedSku?.marketplaces ?? {}` корректно. Но в шаге 3 `patchSkuMarketplace` делает `const next = { ...s.marketplaces }` — spread undefined не вызовет ошибки в рантайме, но TypeScript может потребовать `s.marketplaces ?? {}`. Несоответствие между «NOT NULL в БД» и `?` в типе создаёт мелкое несоответствие, которое агент будет замечать в разных местах и будет неуверен, ставить `?? {}` или нет.

**Потеря:** 5-10 минут хождения по шагам 3-4 с вопросом «почему тип optional если поле NOT NULL».

**Исправление:** уточнить в шаге 1 — либо `marketplaces: Partial<Record<Marketplace, string>>` (обязательное, зеркально NOT NULL), либо объяснить почему `?` (например, обратная совместимость с существующими данными без колонки до миграции). Один путь — правило #1.

- Скрытых зависимостей, которые приведут к ошибке при линейном выполнении (1→2→3→4→5), нет.

- Пробелов «нет шага для критичного действия» не обнаружено: миграция, extract, seed, UI, витрина, e2e, деплой-инструкция — всё покрыто.
