# Review Summary — marketplace-sku-links

> Дата: 2026-06-11
> Ревью: code + risks + structure + coverage (sonnet × 4)

## Критичное (блокирует)

1. **Тихое затирание sku-ссылок при сохранении формы** (code #1, risks #4).
   `upsertSkus` будет писать `marketplaces: sku.marketplaces ?? {}` — если
   product-mapper НЕ передаст `sku.marketplaces`, каждое сохранение товара
   молча сотрёт все ссылки (без ошибки, typecheck не поймает — поле optional).
   План это описывает (step_1 §4), но без защиты. **Фикс:** (а) step_1 §4 —
   выделить жирным «без этого пункта сохранение формы стирает ссылки»;
   (б) step_3 e2e += тест-регрессия: заполнить sku-ссылку → сохранить → снова
   открыть и сохранить НЕ заходя на таб «Маркетплейсы» → ссылка на месте.

2. **`cleanMarketplaces` уже имеет подходящую сигнатуру** (code, важное →
   сюда как часть фикса #1). step_1 §4 предписывает «обобщить сигнатуру» —
   лишний рефакторинг, текущая `(m: Product['marketplaces'])` уже принимает
   `Partial<Record<Marketplace,string>> | undefined`. **Фикс:** step_1 §4
   переписать: просто вызвать существующую функцию для sku-уровня.

## Важное

3. **`db:seed` в afterAll (admin.spec, admin-marketplaces.spec) стирает
   sku.marketplaces** (risks #2, structure #2). Новые e2e самодостаточны
   (ссылки создают через UI) — полный прогон стабилен, НО SQL-проверка импорта
   из step_2 после полного e2e покажет 0 и агент будет «чинить» рабочий сидер.
   **Фикс:** step_2 — явная пометка в командах верификации: SQL-проверки
   выполнять СРАЗУ после сидера, ДО любых e2e; после полного прогона —
   перезапустить `db:seed-mp-links`.

4. **`Sku.marketplaces?` (optional) против NOT NULL DEFAULT '{}' в колонке**
   (structure #3, правило #1 — один путь). **Фикс:** read-тип БЕЗ `?`:
   `marketplaces: Partial<Record<Marketplace, string>>` — mapSku всегда
   заполняет из NOT NULL колонки; `?? {}` в читателях не нужен. Write-схема
   остаётся optional (форма может не слать пустую мапу).

5. **Причина зависимости шага 4 от шага 3 сформулирована как «тот же
   спек-файл»** (structure #1) — реальная причина: витринному e2e нужны
   ссылки, записанные тестом шага 3 через таб. **Фикс:** переписать поле
   «Зависит от» шага 4.

6. **`OZON_URL = 'https://www.ozon.ru/...'` в существующем спеке** (code #2):
   продуктовый уровень не валидируется — конфликт мнимый. **Фикс:** step_3 —
   явная пометка «существующие URL-константы спека НЕ менять» (чтобы агент
   не начал „чинить" ozon.ru → ozon.kz).

## Мелочи

- Валидация extract-скрипта (ozon.kz/kaspi.kz) против «10 артикулов уточнить»
  (risks #5): проверено grep'ом по реальному CSV — все 109 строк имеют ссылки
  с правильными префиксами, скрипт не упадёт. Не править.
- `aria-label="Чёрный"` для свотча sv7 (code #3): подтверждено — сид содержит
  colorLabel «Чёрный», паттерн уже работал в inventory-lifecycle. Не править.
- Пересборка tools-образа при обновлении ТОЛЬКО JSON (risks #6): уже покрыто
  «Прод-деплоем» PLAN.md (шаги 2 и 5 оба с --no-cache). Не править.
- 10 артикулов «уточнить» не имеют отдельной ветки в step_2 (coverage):
  сознательно — Out-of-scope. Не править.
