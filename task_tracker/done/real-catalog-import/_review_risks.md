# Review: Risks — Real Catalog Import + CRUD Foundation

> Проверено: PLAN.md, progress.md, step_1..step_8.md, CLAUDE.md проекта и глобальный CLAUDE.md.
> Дата: 2026-05-29

## Критичное (только [CONFIRMED] и [LIKELY])

### 1. TRUNCATE CASCADE без явного подтверждения пользователя [CONFIRMED]

**Шаг 5** (`step_5_import_script.md`, строка 13):

```
TRUNCATE products, product_variants, skus, media_assets, orders, order_items, inventory_log CASCADE
```

Это деструктивная операция сброса данных — попадает под правило "СТОП" из глобального CLAUDE.md:
> "Сбрасывает / обнуляет / удаляет данные (`UPDATE SET NULL`, `DELETE`, `DROP`, reset флагов)"
> "Порядок: 1. Скажи что собираешься сделать... 2. Дождись явного подтверждения. 3. Только потом выполняй."

В плане нет шага ожидания подтверждения перед TRUNCATE. Ralph-агент выполнит `npm run db:seed` автономно, что означает мгновенный TRUNCATE всех таблиц без интерактивного подтверждения. Guard проверяет только имя БД (`tanar_dev|tanar_test`) — не запрашивает "да/нет".

**Риск:** Если в БД к тому моменту появятся тестовые данные от ручных экспериментов — они будут уничтожены без предупреждения. Хуже: TRUNCATE охватывает `orders`, `order_items`, `inventory_log` — таблицы которые план **не трогает** в других шагах, но CASCADE зацепит их автоматически.

### 2. Удаление seed-data.ts необратимо без бэкапа [CONFIRMED]

**Шаг 5** (`step_5_import_script.md`, строки 33-36):

> "Удалить `src/core/db/seed-data.ts`."

В плане нет инструкции сделать git-коммит существующего seed-data.ts **до** удаления. При этом:
- Каждый шаг = отдельный коммит (критерий в PLAN.md, строка 63)
- Если шаг 5 выполнят корректно, файл попадёт в git-историю через коммит удаления — откат возможен через `git show`
- Но агент может удалить файл **до** коммита шага 4 (если выполняет несколько действий в одном шаге), и если что-то пойдёт не так до коммита — файл потерян из рабочего дерева

**Риск:** Невысокий при дисциплинированном "один шаг = один коммит", но нигде явно не написано "сначала закоммить текущее состояние seed.ts, потом удаляй". Для агента-исполнителя это неочевидно.

### 3. Частичный импорт при ошибке на 7-м из 12 товаров [LIKELY]

**Шаг 5**, строка 25: "Для каждого `ProductInput` → `await createProduct(input)`."

`createProduct` (шаг 3) использует транзакцию **на уровне одного товара** (product+variants+skus атомарно). Но весь цикл из 12 товаров — вне общей транзакции. Если падает товар 7:
- Товары 1-6 уже в БД (commit)
- Self-check (строка 26) бросит `throw` при расхождении чисел
- Но `queryClient.end()` вызывается только в строке 31 — **после** self-check

**Итог:** БД остаётся в частично заполненном состоянии (6 из 12 товаров). Повторный `db:seed` исправит это через TRUNCATE в начале (идемпотентность), но только если агент запустит скрипт снова. Если он считает ошибку некритичной или останавливается — БД зависает в невалидном состоянии.

**Подтверждение:** В шаге 5 нет инструкции "при ошибке в цикле — перезапустить". Plan.md критерий идемпотентности (строка 61) это закрывает, но только если агент понимает что "повторный прогон" нужен именно в этом случае.

## Важное (только [CONFIRMED] и [LIKELY])

### 4. Миграция к test-БД — инструкция двусмысленная [CONFIRMED]

**Шаг 1** (`step_1_schema_fields.md`, строки 28-30):

> "Применить к test-БД: `DATABASE_URL="<test-url из .env.local>"` npx drizzle-kit migrate (или через --env-file если test-url в отдельном файле — **проверить как сделано в Фазе 0**)."

"Проверить как сделано в Фазе 0" — отсылка без конкретного ответа. В progress.md (строка 54) написано:
> "env в tsx-скриптах: ТОЛЬКО `tsx --env-file=.env.local -r tsconfig-paths/register`. НЕ dotenv внутри скрипта."

Но это про tsx, а не про `drizzle-kit`. Переменная `DATABASE_TEST_URL` (или как она называется в .env.local) нигде в плане не задокументирована. Агент может подставить неверное имя переменной или передать её неправильным способом для drizzle-kit — миграция пройдёт только к dev, test-БД останется на старой схеме, e2e упадут при запуске с test-БД.

### 5. Зависимость от Docker без явной проверки его состояния [CONFIRMED]

**Шаг 1** (верификация): `npm run db:generate`, `npm run db:migrate` — без предшествующего `npm run db:up`.

**Шаг 5** (строка 50): `npm run db:up` указан в верификации, но не в самой логике шага.

В CLAUDE.md проекта (строка 40) написано:
> "Предусловие для build/test:e2e/dev: `.env.local` с `DATABASE_URL` + поднятый Postgres."

В шагах 1, 2, 3, 4 предусловие `db:up` явно не указано в секции "Реализация" — только иногда в "Верификации". Агент-исполнитель (Ralph) работает между итерациями с перезапуском шелла. Если Docker не поднят между итерациями — `db:migrate` и `db:generate` упадут с Connection refused без явного объяснения что нужно поднять Docker.

### 6. Маппинг care: null vs undefined в ProductInput [LIKELY]

**Шаг 5**, строка 22:
> "`care` — `p.care` (может быть null → передаём undefined или null; в БД nullable)"

Формулировка "undefined или null" — выбор оставлен агенту. В шаге 3 (`step_3_crud_contract.md`) zod-схема валидирует `ProductInput`. Если zod-схема объявляет `care?: string` (optional), то `null` не пройдёт валидацию (zod по умолчанию не принимает `null` там где `optional`). Нужно либо `care?: string | null` с `.nullable()` в zod, либо явно конвертировать `null → undefined` в маппере снапшота.

**Риск:** zod бросит `ZodError` на первом товаре без care, если агент напишет `care: p.care` и p.care === null.

### 7. scripts/check-images.ts — потребитель seed-data вне src/ [CONFIRMED]

**progress.md**, строка 49:
> "`scripts/check-images.ts` — импортит `getAllProducts`/`seed-data` (вне src/!). При удалении seed-data сломается. **Урок Фазы 0: grep потребителей по ВСЕМУ репо, не только src/.**"

**Шаг 5** предусматривает решение (переключить на снапшот/no-op). Риск не в самом факте поломки, а в том, что если агент выполняет шаги 1-4 без удаления seed-data.ts, а потом на шаге 6 запускает `npm run typecheck` или `npm run lint` — check-images может не проверяться до шага 5. Критерий "Grep `seed-data` по всему репо → 0" стоит только в шаге 5. Если агент пропустит fix check-images.ts — `images:check` сломается незаметно до финального шага 8.

### 8. Доменный тип price vs priceBase — риск несоответствия при updateProduct [LIKELY]

**progress.md**, строка 58-59:
> "Доменный тип `Product.price` (не priceBase) — repository mapper кладёт `row.priceBase → price`. При write-контракте: вход zod-схемы должен мапить обратно `price/priceBase`. Решить единообразно в шаге 3."

**Шаг 3** фиксирует: "Имя поля цены во входе — `priceBase`". Но `updateProduct` вызывает `getProductBySlug(slug)` в конце для возврата. Если read-mapper возвращает `price` (не `priceBase`), а вызывающий код (импорт-скрипт) проверяет `result.priceBase` — это undefined. Скрипт не упадёт (undefined !== Error), но self-check по числам не поймает это несоответствие.

В шаге 5 self-check проверяет только количество записей (12/30/109), не содержимое полей. Цена может быть записана неверно и это не обнаружится до визуальной проверки витрины.

### 11. catalog-snapshot.json перемещается в done/ вместе с папкой плана [THEORETICAL]

Шаг 8 финального чеклиста (строка 30): "вместе с catalog-snapshot.json — он остаётся снапшот-источником для Фазы 5". После переезда в `done/` импорт-скрипт (seed.ts) будет читать файл по относительному пути `task_tracker/todo/...` — который после переезда станет `task_tracker/done/...`. Скрипт упадёт при следующем `db:seed`. Но это произойдёт уже после завершения плана, при первом запуске в новой сессии. Теоретически возможно, практически — агент должен обновить путь в seed.ts или скопировать/переместить снапшот в другое место.

## Не найдено проблем

- **Граница модулей (ESLint):** план явно запрещает write в client.ts, импорты через index.ts — контролируется lint-критериями в каждом шаге.
- **Порты Postgres 5442/5443:** задокументированы в progress.md как КРИТИЧНЫЕ факты, guard в seed.ts проверяет имя БД.
- **Идемпотентность:** TRUNCATE в начале скрипта обеспечивает идемпотентность при корректном прогоне — критерий в PLAN.md строка 61.
- **Ozon-поля не попадают в core:** явно зафиксировано в шаге 5 строка 22 и критерии шага.
- **public/images/ не трогается:** явно в "Что НЕ делать" и критерии шага 8.
- **Транзакции на уровне одного товара:** корректно для createProduct; updateProduct с заменой variants целиком — обоснованное решение зафиксировано в шаге 3.
