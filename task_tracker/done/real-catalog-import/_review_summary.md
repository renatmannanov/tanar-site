# Review Summary — Real Catalog Import (План A)

> Дата: 2026-05-29
> Ревью: code + risks + structure (3 агента, sonnet)

## Критичное (блокирует / даёт неверный результат)

1. **ProductCard покажет битый `<Image>` вместо градиента для ВСЕХ боевых товаров** (code #4).
   В снапшоте `variants[].models: []`. `ProductCard.tsx` зовёт `getProductCardImage`, который возвращает объект всегда (не null) → `src` вида `front-undefined-card-md.webp`, файла нет → битая картинка на всей витрине. План этого НЕ покрывает.
   → Нужен шаг/правка: guard `models.length > 0` (или `hasFlatShots`) перед показом фото, иначе fallback на градиент (`Placeholder`). Проверить ту же логику в `ProductDetail`/галерее.

2. **PowerShell vs bash для миграции к test-БД** (code #3, risks #3, structure #1).
   step_1 даёт `DATABASE_URL="..." npx drizzle-kit migrate` — это bash, в PowerShell не работает. Плюс две формулировки через «или» («или через --env-file») = автономный агент выберет наугад. Имя `DATABASE_TEST_URL` нигде не зафиксировано.
   → Зафиксировать ОДНУ команду: `$env:DATABASE_URL = "<test-url>"; npx drizzle-kit migrate` (PowerShell), указать точное имя переменной из .env.local.

3. **TRUNCATE CASCADE автономно — против правила «СТОП перед деструктивными»** (risks #1).
   Импорт-скрипт TRUNCATE'ит в т.ч. `orders/order_items/inventory_log` (план их не трогает). Сейчас они пусты, но правило проекта требует осторожности. Для Ralph — приемлемо (dev-БД, guard на tanar_dev/test), но стоит явно сузить TRUNCATE до каталожных таблиц ИЛИ задокументировать, что orders пусты в Плане A и это ок.
   → Решение: TRUNCATE только `products, product_variants, skus, media_assets CASCADE` (каталог). orders не трогаем.

## Важное

4. **`care: null` уронит zod на первом товаре без care** (risks #5).
   Снапшот: `care: null` у 7 товаров. zod `.optional()` не принимает `null`. `import → createProduct → productInputSchema` упадёт ZodError.
   → В zod-схеме: `care: z.string().nullable().optional()`; в мэппинге `care: p.care ?? undefined`. Зафиксировать в step_3 (схема) и step_5 (маппинг).

5. **`check-images.ts` — однозначно зафиксировать действие** (code #2, risks #6, structure #4).
   step_5 пишет «переключить на снапшот / no-op» — два разных объёма. Файл вне `src/`, typecheck не ловит, tsx упадёт в рантайме после удаления seed-data.
   → Зафиксировать ОДИН вариант: переключить на чтение `catalog-snapshot.json`; при пустом списке фото печатать `images:check: no product photos yet` и exit 0. Убрать «/no-op» из формулировки.

6. **Footer/CategoriesGrid — двойной источник правды step_2 vs step_6** (structure #3).
   step_2 «минимально поправить ради typecheck», step_6 даёт финальные значения. Агент не знает, что подставить временно.
   → Убрать минимальную правку из step_2: вместо неё временно закомментировать/заменить на `jackets` (валидный во всех версиях). ЛУЧШЕ: объединить — step_2 правит ТОЛЬКО типы/категории/contracts, а Footer/CategoriesGrid правятся ЦЕЛИКОМ в step_6, а между ними typecheck допускает ошибку (но коммит step_2 тогда не «зелёный»). Альтернатива: перенести правку Footer/CategoriesGrid финальными значениями в step_2 целиком, step_6 их уже не трогает. → Выбрать: **финальные значения Footer/CategoriesGrid делаем в step_2** (раз уж типы там меняются), step_6 оставляет только metadata + проверку [slug]/build. Это убирает двойной источник.

7. **`db:up` как предусловие шагов 1-5, не только верификации** (risks #4).
   Между итерациями Ralph Docker может быть не поднят. db:migrate/db:seed упадут с auth-ошибкой (см. Фазу 0).
   → В step_1 и step_5 добавить `npm run db:up` первой командой реализации.

8. **`zod` поставить первым действием step_3** (code #1) — уже в плане, но усилить: `npm i zod` ДО написания кода схемы.

9. **self-check импорта проверяет только количество, не цены/поля** (risks #7).
   12/30/109 совпадёт даже если `priceBase`/`article` легли неверно.
   → Добавить в self-check点 проверку 1-2 известных значений (напр. `jacket-sv7-goretex` priceBase=80000; один article=TANAR-001 существует). Дёшево, ловит ошибки маппинга.
