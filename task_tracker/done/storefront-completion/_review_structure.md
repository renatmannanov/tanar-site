## Важное (исполнитель потеряет >5 минут)

### 1. step_3 и step_4 — пересечение по файлу `catalog/[slug]/page.tsx`

PLAN.md заявляет шаги 3 и 4 независимыми. Но:

- **step_3** переключает `src/app/(public)/catalog/[slug]/page.tsx` с `getProductBySlug` на `getStorefrontProductBySlug` (и добавляет `notFound()`).
- **step_4** явно не трогает эту страницу — он правит `ProductDetail.tsx` и `ProductCard.tsx`.

Прямого конфликта нет, но если исполнитель делает шаги 3 и 4 одним коммитом или в разном порядке — взаимодействия нет. Риск: шаг 4 добавляет поля в `ProductDetail.tsx`, а шаг 3 добавляет `notFound()` в page-обёртку. Если шаг 4 выполнить ДО шага 3 и запустить e2e — тест «draft → 404» провалится (функция ещё не переключена). Это не ошибка в данных, но может сбить с толку при пошаговой верификации.

**Рекомендуемое уточнение:** добавить в step_4 примечание «если шаг 3 уже выполнен — ` getStorefrontProductBySlug` уже на месте; если нет — e2e по статусам запускать только после шага 3».

Потеря времени: ~10 минут на разбор «почему тест статуса проваливается на шаге 4».

### 2. step_2 — развилка «фильтровать ИЛИ положиться на zod»

Текст step_2 (секция «При submit»):
> «Зафиксировано: фильтровать пары, где `label.trim()===''` в onSubmit»

Формулировка «Зафиксировано» после «ИЛИ» оставляет лёгкую неоднозначность: сначала предлагается два варианта, затем один объявляется зафиксированным в той же фразе. Автономный агент может прочитать первую часть предложения как открытую развилку и остановиться.

Критичности нет (зафиксированный вариант явно указан), но форма записи создаёт паузу. Потеря: ~5 минут на перечитывание.

---

### 3. step_5 — предусловие `npm run build` перед e2e: порядок шагов не указан

step_5, «Команды для верификации»:
```
npm run db:up; npm run db:seed
npm run build
npm run test:e2e
```

При этом progress.md содержит критичную грабли:
> «НЕ `npm run build` пока жив `npm run dev`» — build затирает `.next` → dev падает ENOENT.

В step_5 нет явного «остановить dev перед build». Все предыдущие шаги (1-4) заканчиваются на `npm run dev` (ручная проверка). Если агент прогнал dev-проверку в шаге 4 и не остановил dev — build в шаге 5 сломает dev-сессию незаметно, а затем e2e упадут по инфраструктурной причине, не по коду.

**Уточнение:** добавить строку `# убедиться что dev остановлен` перед `npm run build` в step_5.

Потеря: ~10-15 минут на диагностику падения e2e по инфраструктурной причине.

## Мелочи (формальные)

### 4. step_3 — фраза «Можно вынести общий хелпер фильтра, но НЕ обязательно»

Формально это развилка в инструкции, но сразу же зафиксирована: «зафиксировано: просто добавить 4 функции с inArray-условием». Агент прочитает зафиксированную часть и пойдёт дальше. Не критично.

### 5. step_4 — «рядом с названием (h1) или ценой»

Текст: «рядом с названием (h1) или ценой — `product.label?.badge`». Технически развилка по вёрстке, но в progress.md зафиксировано «рядом с названием/ценой» как допустимые позиции. Для автономного агента это свобода выбора, а не блокер — результат будет корректным при любом выборе. Некритично.

### 6. step_6 — упоминание `ARCHITECTURE-ecommerce.md`

Шаг 6 чеклист: «ARCHITECTURE-ecommerce.md: отметить, если релевантно». Этот файл нигде не упомянут в PLAN.md и progress.md. Если файла не существует — агент потратит время на поиск. Условие «если релевантно» снижает риск, но путь к файлу не указан.
