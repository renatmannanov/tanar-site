## Критичное (только [CONFIRMED] и [LIKELY])

### 1. Slug-коллизия при автогенерации → unique violation в БД [CONFIRMED]

**Шаг 1** (step_1_slug_autogen.md).

Slug хранится с `UNIQUE` constraint (`products_slug_unique` — подтверждено в `src/core/db/schema.ts:21` и миграции `0000_same_typhoid_mary.sql:74`). `createProduct` в `repository.ts:401-415` не обрабатывает нарушение уникальности: `tx.insert(...).values(...)` бросит Postgres-ошибку `23505`, которая не перехвачена — пользователь получит необработанный 500.

**Сценарий:** два товара с похожими названиями, например «Куртка SV7» и «Куртка SV8» — оба дадут `kurtka-sv7` и `kurtka-sv8`, коллизии нет. Но «Куртка» и «Куртка» → оба `kurtka` → unique violation при create второго. Или название из одних цифр/латиницы, совпадающее со slug уже существующего товара.

**Что план говорит:** шаг 1 указывает только `slugify('Куртка SV7 Gore-Tex®') === 'kurtka-sv7-gore-tex'` как happy path. Обработка коллизии — ни в шаге 1 (`step_1_slug_autogen.md`), ни в progress.md — **не упомянута вообще**.

**Последствие:** при создании второго товара с тем же slug — 500 без диагностики для пользователя. Для e2e-теста в шаге 5: если `db:seed` не был выполнен после предыдущего прогона, тест-товар с slug `testovaya-kurtka-x1` может уже существовать — тест упадёт с 500 вместо ожидаемой навигации.

**Минимальный фикс (не требует изменения плана):** в server action создания поймать Postgres error code `23505` и вернуть пользователю сообщение «Товар с таким slug уже существует — измените название» вместо 500.

### 2. Пустой slug при name из одних спецсимволов → zod min(1) отвергает с 500 [CONFIRMED]

**Шаг 1** (step_1_slug_autogen.md строки 17, 27).

Сам шаг это упоминает: «Если на выходе пусто — вернуть пустую строку (форма не даст создать с пустым slug — zod min(1))». Но «не даст создать» работает только если форма валидирует slug на клиенте ДО submit. В ProductForm сейчас slug — управляемое поле `readOnly`, но валидации slug до submit в форме нет. Итог: поле пусто → submit → `productInputSchema.parse` в server action бросает zod ValidationError → 500.

Zod schema `slug: z.string().min(1).regex(...)` подтверждена в `repository.ts:204-207`.

**Конкретный кейс:** `slugify('®™ / ')` → после обработки всё → дефисы → схлопывание → trim → пустая строка. Пользователь видит поле slug пустым, кнопка «Создать» визуально не заблокирована (нет client-side валидации поля slug), нажимает — 500.

**Минимальный фикс:** в ProductForm дизейблить кнопку «Создать» когда `form.slug === ''` (или `form.slug.trim() === ''`).

## Важное (только [CONFIRMED] и [LIKELY])

### 3. generateMetadata в product page использует старый getProductBySlug — не учтён в шаге 3 [CONFIRMED]

**Шаг 3** (step_3_status_visibility.md строка 28).

Шаг явно говорит: «`getProductBySlug` → `getStorefrontProductBySlug` (в `generateMetadata` И в `ProductPage`)». Это правильно. Риск — агент может переключить только основной render (`ProductPage`), но забыть `generateMetadata`. В таком случае: draft-товар по прямому URL → `generateMetadata` вернёт заголовок (не 404), но `ProductPage` вызовет `notFound()`. Это приведёт к inconsistency: страница 404, но `<title>` (в head) при SSR установится заголовком товара.

**Почему это риск для агента:** шаг 3 упоминает это правило в одном предложении без выделения. Прецедент из предыдущих планов (Plan B/C) — агент упускал детали при нескольких изменениях в одном файле.

**Тест из шага 5 это не ловит:** тест проверяет `response.status()===404`, но не проверяет содержимое `<title>`.

### 4. Новые storefront-функции не экспортированы через index.ts → ESLint-ошибка или silent miss [LIKELY]

**Шаг 3** (step_3_status_visibility.md строка 22).

Шаг указывает: «Экспортировать из `@/core/catalog` (index уже `export * from repository`)». Это правильно. Риск — если агент добавляет функции, но не проверяет, что `index.ts` действительно делает `export * from './repository'` (а не именованные экспорты). Если `index.ts` использует именованные экспорты, новые функции нужно добавить явно.

Проверить: `src/core/catalog/index.ts` — паттерн экспорта.

**Последствие при miss:** витринные страницы не найдут функцию на импорте → `build`-ошибка (ловится `npm run build`). Но если агент проверяет только `typecheck`/`lint` и пропускает `build` — дефект уйдёт в коммит.

**Примечание:** шаги 1 и 3 явно требуют `npm run build` в критериях. Если агент соблюдает критерии — риск закрыт. Но прецедент из Plan C: build-проверка иногда пропускается при «очевидных» изменениях.

### 5. E2E-тест шага 5: hardcoded slug 'testovaya-kurtka-x1' совпадёт с именем «Тестовая Куртка X1» только если транслит реализован конкретно [LIKELY]

**Шаг 5** (step_5_e2e.md строка 15).

Тест проверяет: поле `#slug` стало `testovaya-kurtka-x1`. Однако конкретный транслит «е→e», «я→ya», «к→k» нигде в плане не зафиксирован как единственно правильный. Альтернативные схемы дают разные результаты: «т» → `t`, «е» может быть `ye` (ISO), «к» → `k`, «я» → `ia` (старая схема). Итог: реализация агентом таблицы транслита может не совпасть с ожиданием теста.

**Фикс в плане (не требует изменения архитектуры):** либо в шаге 5 использовать `expect(slugValue).toMatch(/^testovaya-kurtka-x1/)` как вариант, либо зафиксировать точную таблицу транслита в `step_1` и повторить ожидание в `step_5`. Сейчас шаг 1 даёт таблицу частично («а→a, б→b, в→v... е→e, ё→e»), но «я→ya» прямо указано (строка 14). Для «Тестовая Куртка X1»: т→t, е→e, с→s, т→t, о→o, в→v, а→a, я→ya → `testovaya`. Если агент следует таблице из шага 1 буква-в-букву — совпадёт. Но риск ненулевой при отклонении.

### 6. Шаг 5 test п.2 (specs-редактор) не может верифицировать сохранение без перезагрузки страницы [LIKELY]

**Шаг 5** (step_5_e2e.md строка 16-17).

Тест: «добавить характеристику "Материал"/"Нейлон" → Сохранить → вернуться на edit → характеристика на месте». Если агент реализует проверку через «вернуться на edit» — это корректная end-to-end проверка через round-trip в БД. Но тест описан с `или проверить на витрине после publish` как опция. Если агент выбирает путь «проверить на витрине» — нужен publish перед проверкой, иначе витрина вернёт 404 (товар в draft). Шаг 5 не фиксирует жёстко какой путь использовать → риск невалидного теста.

## Мелочи ([THEORETICAL])

### T1. Активный вариант без skus — секция «Размеры» не показывается [THEORETICAL]

Шаг 4 явно говорит: «Если у активного цвета нет skus — секцию не показывать». Это корректный guard. Риск только теоретический: если агент забудет guard — секция отрендерится пустой (без чипов), что некритично визуально, но UX-мусор. Тест шага 5 проверяет боевой товар `jacket-sv7-goretex`, у которого skus есть → тест не поймает этот кейс. Риск незначительный (нет данных в БД без skus для published товаров).

### T2. coming_soon: бейдж label.badge и бейдж «Скоро» в ProductCard — только для listed [THEORETICAL]

Шаг 4 (строка 18) явно говорит: «если coming_soon → показываем только "Скоро", иначе → label.badge если есть». Это зафиксировано как решение. Риск только если агент неверно понимает приоритет — но инструкция однозначна.

### T3. care с переносами строк: `whitespace-pre-line` и Tailwind purge [THEORETICAL]

Шаг 4 упоминает `whitespace-pre-line`. Это стандартный Tailwind-класс, он включён через content-glob в `tailwind.config.ts`. Теоретически может быть вырезан при purge если не встречается в контенте. Практически — `whitespace-pre-line` — стандартный utility, не динамический → не purge-уязвим.

## Не найдено проблем

- **Необратимые операции (DROP/TRUNCATE/DELETE данных):** план не содержит. `db:seed` в afterAll — только `INSERT` через createProduct после `db:reset`. Не деструктивно для продакшена.
- **Нарушение правила `@/core/catalog/client`:** шаги правят `ProductForm.tsx` (client) и `repository.ts` (server). Новые storefront-функции идут в `repository.ts` → экспортируются через `@/core/catalog` → витринные страницы (server components) их используют. Нарушения клиент/серверной границы не предполагается.
- **`slugify` как client-safe:** шаг 1 явно требует «без node/server-импортов». Функция — чистая (lookup table + string ops). Риска нет при следовании инструкции.
- **Совместное редактирование ProductForm.tsx шагами 1 и 2:** план явно говорит «выполнять 1 → 2 последовательно» и повторяет это в обоих шагах. Риск потери правок при параллельном выполнении закрыт инструкцией.
- **db:seed в afterAll e2e:** шаг 5 явно прописывает `afterAll: db:seed`. Тестовые данные не останутся в БД.
