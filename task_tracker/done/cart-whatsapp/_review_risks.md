# Review: Risks

## Критичное (только [CONFIRMED] и [LIKELY])

### 1. ALTER TABLE на продакшен-БД с данными — нет отката [CONFIRMED]
**Шаг 1.** `orders.number` добавляется как `serial('number').notNull()`. При `ALTER TABLE orders ADD COLUMN number serial NOT NULL` Postgres автоматически присваивает значения существующим строкам (auto-backfill) — это необратимо. На проде уже есть живая БД (VPS 89.219.32.75), и если в момент миграции в таблице окажутся заказы с `number IS NULL`, а логика чего-то не ожидала, откат (`ALTER TABLE orders DROP COLUMN number`) потеряет присвоенные номера.

Конкретнее: план не упоминает проверку `count(orders)` перед миграцией и не предусматривает `db:reset` на проде перед деплоем. Это приемлемо (serial backfill стандартен), **но** шаг 1 не говорит агенту проверить, что на проде нет заказов с NULL-номером после миграции. Если деплой пройдёт, а в `createOrder` вернётся строка без `number` (тип расходится до применения шага 1) — номер заказа на экране подтверждения будет `undefined`.

**Ссылка:** `step_1_db_orders_module.md:16-20`, `src/core/db/schema.ts:178`.

### 2. `ProductDetail.tsx` — клиентский компонент; передать `whatsapp` невозможно без изменения вызывающей страницы [CONFIRMED]
**Шаг 4.** В `catalog/[slug]/page.tsx` вызывается `getSiteSettings()` (шаг 4, §1). Но `ProductDetail.tsx` — `'use client'` (строка 1 файла). Значение `whatsapp` — результат server-side запроса к БД; его нужно пробросить через props из server-компонента страницы в клиентский `ProductDetail`. Это корректный паттерн, **НО**: `ProductDetail` уже принимает props типа `Product + shots + ...`; нужно убедиться, что добавление `whatsapp?: string | null` в его prop-интерфейс не конфликтует с тем, что props-тип сейчас определён (нет экспортированного интерфейса — разные шаги могут определить его по-разному).

Отдельный риск: если агент забудет добавить `whatsapp` в prop-тип `ProductDetail` до обращения к нему внутри компонента — typecheck упадёт, и агент должен будет исправить это итеративно. Шаг 4 описывает это кратко: «принять `whatsapp`» — без явного указания обновить сигнатуру `ProductDetail`. Двусмысленность.

**Ссылка:** `step_4_product_add_to_cart.md:12-15`, `src/components/product/ProductDetail.tsx:1`.

### 3. Дедуп по `cartHash` — нет защиты от гонки (двойного сабмита) [LIKELY]
**Шаг 6.** Дедуп реализован через `localStorage`: до вызова action клиент читает `loadLastOrder()`, сравнивает хэш. Но: между двумя кликами «Оформить» (случайный двойной клик, или пользователь нажал кнопку, action завис, нажал ещё раз) action может вызваться дважды до того, как первый вернул результат и сохранил `lastOrder`. Результат — два заказа с одним и тем же составом и разными номерами в БД.

На сервере нет unique constraint или advisory lock, предотвращающего такую гонку. Шаг 6 описывает только `pending`-стейт кнопки UI (`disabled` во время вызова), но `pending`-стейт в Server Actions / React 19 работает только при одном вызове — дополнительный клик до получения ответа всё равно пошлёт запрос.

Риск умеренный (MVP, WhatsApp-переписка всё равно уточнит дубли), но в плане нет явного требования к серверному дедупу.

**Ссылка:** `step_6_checkout.md:45-50`.

## Важное (только [CONFIRMED] и [LIKELY])

### 4. `coming_soon` товаров нет в сиде — e2e тест из шага 4 потребует создать его через UI [CONFIRMED]
**Шаг 4, тесты.** `seed.ts` явно комментирует: `// status omitted → defaults to 'published'` — все 12 товаров `published`. Тест на `coming_soon`-ветку описан в шаге 4: «В seed-каталоге найти/обеспечить coming_soon товар; если в сиде его нет — тест создаёт его через админку». Это значит, что e2e для этого кейса зависит от работающего CRUD-сценария в том же тест-прогоне, что добавляет хрупкость (тест не изолирован, зависит от другого шага). Агент скорее всего создаст товар через UI в e2e, но если `admin-crud-media.spec.ts` упадёт — упадут и cart-тесты.

**Ссылка:** `step_4_product_add_to_cart.md:72-74`, `src/core/db/seed.ts:80`.

---

### 5. `serial` не импортирован в `schema.ts` — агент должен добавить импорт [CONFIRMED]
**Шаг 1.** Текущий импорт в `src/core/db/schema.ts:1-11` не включает `serial`. Шаг 1 прямо говорит: «(импорт `serial` из `drizzle-orm/pg-core`)» — значит, агент ДОЛЖЕН добавить его в импорт вручную. Если забудет — `npm run db:generate` сгенерирует некорректную или сломанную миграцию, TS-ошибка поймается только на typecheck.

Риск управляем (typecheck ловит), но создаёт итеративный цикл fix→typecheck если агент пропустит.

**Ссылка:** `step_1_db_orders_module.md:16`, `src/core/db/schema.ts:1-11`.

---

### 6. WhatsApp URL length — длинный заказ может превысить лимит wa.me [LIKELY]
**Шаг 6.** `wa.me` click-to-chat принимает `?text=` в URL. Браузеры и мессенджеры имеют лимит ~2000 символов на URL. При максимальной корзине (30 позиций × длинные названия + `nameSnapshot` = «{name} ({colorLabel}, {size}/{ruSize})») + URL-encoding (×3) итоговый URL может превысить лимит. WhatsApp Web и мобильный app ведут себя по-разному при обрезании.

Шаг 6 не описывает обрезку текста или fallback при слишком длинном URL. Для MVP (реальные заказы небольшие) риск низкий, но не нулевой.

**Ссылка:** `step_6_checkout.md:26-36`.

---

### 7. Двусмысленность: шаг 5 — кнопка оформления `disabled` с `title="Скоро"` — шаг 6 «заменяет содержимое подвала» [CONFIRMED]
**Шаги 5→6.** Шаг 5 говорит: «кнопка `disabled` с `title="Скоро"`; шаг 6 включает её, API уже финальный (рядом onClick-заглушка не нужна — шаг 6 заменяет содержимое подвала на двухэкранный flow)». Шаг 6 говорит: `CheckoutPanel.tsx` используется в подвале `CartDrawer`, «заменяет disabled-кнопку шага 5».

Агент, работающий по шагу 5, может добавить локальный `onClick`-обработчик как заглушку. Шаг 6 потом заменит его `CheckoutPanel`. Если агент в шаге 5 вынесет кнопку в отдельный компонент — шаг 6 легко подменит его. Если встроит inline — шаг 6 потребует редактировать CartDrawer в том же месте, где шаг 5 только что всё написал, что увеличивает риск конфликта.

Инструкция «API уже финальный (onClick-заглушка не нужна)» может быть неправильно прочитана как «не делать ничего», хотя имеется в виду «не пиши stub, просто `disabled`».

**Ссылка:** `step_5_cart_drawer.md:30-35`, `step_6_checkout.md:53-68`.

### 8. Тест дедупа: `localStorage['tanar-cart-order']` — ключ в шаге 6 называется `tanar-cart-order`, в шаге 3 — нет [CONFIRMED]
**Шаги 3 и 6.** Шаг 3 объявляет `CART_STORAGE_KEY = 'tanar-cart'` для корзины. Шаг 6 вводит **второй** ключ `tanar-cart-order` (`saveLastOrder`/`loadLastOrder`) для дедупа. Эти функции должны быть добавлены в `src/lib/cart.ts` (шаг 6, §2: «В `src/lib/cart.ts` добавить...»). Но шаг 3 создаёт файл `cart.ts` — агент шага 6 ДОЛЖЕН дополнить существующий файл, а не создать новый. Если агент создаст отдельный файл (например, `src/lib/order-storage.ts`), типы будут дублированы или разбиты между файлами, что нарушит правило «CartItem — один, в `src/lib/cart.ts`».

**Ссылка:** `step_3_cart_core.md:25`, `step_6_checkout.md:44-49`.

## Мелочи

### 9. Верификационная команда в шаге 1 ищет `'"number" serial'` в SQL [THEORETICAL]
`grep -l '"number" serial' src/core/db/migrations/*.sql` — drizzle-kit может сгенерировать `"number" serial NOT NULL` (без пробела после `serial`, или иной форматинг). Команда использует `-l` (files with matches) но строка может не совпасть буквально. Агент проверит вручную если получит пустой результат.

### 10. QR-код генерируется в `useEffect` (client-side) [THEORETICAL]
`toDataURL(waUrl)` — асинхронная функция, результат устанавливается в state. На медленных устройствах QR появится с задержкой. Это не баг, но e2e тест на `wa-qr` `<img src^="data:image/">` должен ждать появления атрибута `src`. Если Playwright не поставит `waitFor` — тест будет флакующим.

**Ссылка:** `step_6_checkout.md:66-67`.

## Не найдено проблем

- Необратимые DROP/TRUNCATE/DELETE — отсутствуют. `ALTER TABLE ADD COLUMN` с serial — обратимо (можно дропнуть колонку без потери данных).
- Нарушений правил PII — нет. WhatsApp-номер пользователя не сохраняется в БД (только номер продавца в `site_settings`). Состав заказа в `nameSnapshot` — публичные данные каталога.
- Нарушений правила «client НЕ импортирует server barrel» — план явно оговаривает это в progress.md и каждом шаге где это применимо (шаги 5, 7).
- Критических зависимостей от внешних систем — нет. `wa.me` — просто ссылка; WhatsApp Business API отклонён; `qrcode` — npm-пакет без сетевых вызовов.
- Удаление `AvailabilityButton` — не сломает e2e: `e2e/product.spec.ts` проверен — `availability-button` testid там НЕ используется (файл содержит только тест на 404 для несуществующего slug).
