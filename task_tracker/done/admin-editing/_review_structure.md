# Review: Structure — admin-editing

## Критичное (исполнитель потеряет часы или неверный результат)

### 1. step_3: нерешённая условная развилка — витринный Header/Footer в админке

**Где:** step_3_admin_shell.md, секция «Витринный Header/Footer в админке» (конец файла).

**Что не так:** Написано "Проверить визуально... Если шапка мешает — вынести витрину в route-group `(public)/`". Это условная ветка "выполнять ТОЛЬКО если мешает". Для автономного агента это означает: дойти до шага 3, собрать shell, визуально убедиться, принять архитектурное решение (трогать ли все витринные пути или нет), а затем либо делать крупную правку (перемещение `page.tsx / catalog/ / blog/` в `(public)/`) либо не делать ничего. Объём отличается кардинально: "не делать" = 0 файлов, "делать" = перемещение 10+ файлов с проверкой всех витринных маршрутов.

В criteria шага зафиксировано: "Визуально: админка без outdoor-шапки витрины (**или зафиксировано решение по (public)/)**" — т.е. критерий готовности двузначен: либо сделано, либо задокументировано. Агент может закоммитить шаг с витринной шапкой в админке, написать в Learnings "оставили как есть", и план будет считаться выполненным — но при следующем запуске другой агент увидит шапку и начнёт переделывать. Это разные финальные состояния кодовой базы.

**Что теряет исполнитель:** Если агент решит делать route-group рефакторинг, это 30–60 минут + риск сломать витринные маршруты + повторный прогон 39 e2e. Если не решит — последующий Plan C-агент может сделать иначе.

**Рекомендация:** зафиксировать ОДНО решение до начала исполнения. Либо "делаем route-group (public)/ в шаге 3 безусловно", либо "НЕ делаем — корневой layout остаётся, админка живёт с витринной шапкой до Plan C". Текущая формулировка оставляет выбор агенту по факту визуальной проверки.

### 2. step_3: нерешённая развилка auth-guard в admin/layout.tsx

**Где:** step_3_admin_shell.md, секция «Layout», абзац с тремя «Вариант»/«Решение»/«Финальное решение».

**Что не так:** В тексте шага есть рассуждение с несколькими вариантами и промежуточными "Решение:" перед финальным "Финальное решение:". Финальное решение действительно зафиксировано: "НЕ делать guard в admin/layout.tsx; полагаться на middleware как единственный барьер; requireAdmin() в page-компонентах разделов." Это конкретное решение.

**НО:** PLAN.md в секции «Архитектурные решения» говорит ПРОТИВОПОЛОЖНОЕ: "layout.tsx = auth-guard (серверная проверка cookie дублирует middleware — defense in depth)". Т.е. PLAN.md — guard в layout, step_3 — guard НЕ в layout, а в page. Это прямое противоречие между двумя источниками правды для одного и того же файла.

Агент читает `CLAUDE.md → PLAN.md → step_N.md`. Если агент строго следует PLAN.md при расхождении с step — он положит guard в layout и получит конфликт с login-страницей, который step_3 уже разобрал и решил. Если следует step_3 — нарушит PLAN.md. В любом случае неопределённость.

**Что теряет исполнитель:** Отладка конфликта guard-в-layout + login-redirect, диагностика — 15–30 минут.

## Важное (исполнитель потеряет >5 минут)

### 3. step_6: скрытая зависимость — requireAdmin импортируется из lib, но lib-файл меняется в шаге 3

**Где:** step_6_edit_save.md, строка `await requireAdmin()` в `updateProductAction`.

**Что не так:** `requireAdmin()` описан как функция в `src/lib/admin-auth.ts` — этот файл создаётся в шаге 1 (модуль сессии) и **дополняется** в шаге 3 (добавляется `requireAdmin`). Шаг 6 зависит от шага 5, в зависимостях написано "шаг 5". Но фактически зависит и от шага 3 (где появляется `requireAdmin`). Если шаги выполняются по порядку 1→2→3→…→6 — проблемы нет. Но зависимость в шапке шага 6 неполная — может сбить при попытке параллельного или выборочного выполнения.

**Что теряет исполнитель:** Минимально — 5–10 минут на поиск откуда брать `requireAdmin` если порядок нарушен.

### 4. step_7: тест идемпотентности edit→save — описан как "альтернатива", не зафиксирован способ

**Где:** step_7_e2e.md, пункт 6 «редактирование→сохранение».

**Что не так:** Написано "Затем вернуть исходное (повторный edit или вызвать через UI) — чтобы тест был идемпотентен... Альтернатива: тест меняет и сразу возвращает в рамках одного прогона." Два варианта не зафиксированы в один. Агент может написать тест любым способом. Оба рабочие, но "повторный edit через UI" в рамках одного теста — это 2 последовательных edit-цикла, "сразу возвращает" — 1 цикл с откатом. Если следующий агент (Plan C) перечитает spec и увидит реализацию первого варианта, он не сломает тест, но может запутаться при расширении.

**Что теряет исполнитель:** 5–10 минут на выбор подхода + потенциальный нестабильный тест при первой реализации.

## Мелочи (формальные)

### 5. PLAN.md: критерий "12 боевых товаров" требует db:seed перед e2e

В критериях готовности PLAN.md написано "/admin/catalog показывает 12 боевых товаров". В step_7 предусловие — `npm run db:up; npm run db:seed`. Это согласовано, но нигде не сказано явно: если тест-прогон идёт на чистой БД без seed — критерий провалится. Для шага 7 это задокументировано, для step_8 (финальный smoke) — предусловие не повторено. Несущественно если агент следует порядку, но при изолированном прогоне step_8 может упасть.

### 6. step_1: login-страница — "клиентская обёртка **либо** useActionState"

**Где:** step_1_auth_session.md, секция «Страница логина»: "серверный компонент с формой (action={loginAction} через клиентскую обёртку для показа ошибки, **либо** useActionState)".

Это мягкая развилка реализации — оба варианта технически правильны, различие минимально. Агент выберет сам без потери времени. Не критично.
