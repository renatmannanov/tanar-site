## Цель

Выкатить текущую витрину (Фаза 1.5, готова) на прод — VPS + Docker Compose (web + postgres + caddy, авто-SSL). Фундамент делаем правильно сразу (named-volume для фото и БД, SSL, переносимость), чтобы не переделывать.

**Стратегия запуска (2 этапа):**
1. **Демо на существующем VPS Hetzner** (шаги 1–7) — показать заказчице живой рабочий вариант.
2. **Боевой переезд на PS.kz** (шаг 11) — после демо и оплаты сервера заказчицей. Локально в КЗ. Архитектура к переезду готова (Docker + volumes), код не меняется.

Бэкапы и CLI массовой заливки фото — Фаза deploy-2; бэкап (шаг 8) — это ещё и инструмент переезда (шаг 11).

## Архитектурные решения (зафиксированы с пользователем, без развилок)

- **Деплой:** VPS (PS.kz, Ubuntu 24.04) + Docker Compose. Рантайм-сервисы: `web` (Next.js standalone), `postgres` (volume `pgdata`), `caddy` (reverse-proxy + авто-SSL). One-off `tools`-сервисы (`migrate`/`seed`/`push-media`, profile `tools`) — **все определены в шаге 2** (образ = builder-стадия); шаги 5/6/9 их только запускают, compose не правят.
- **Хранилище фото:** ЛОКАЛЬНЫЙ ДИСК на named volume `product-images` → `/app/public/images/products`. НЕ S3/R2 — намеренно. **Без volume редеплой сотрёт фото — главное прод-требование.** Права volume: web стартует через **entrypoint, chown'ащий папку от root → запуск от node** (volume монтируется root'ом поверх образа; chown в образе не помогает).
- **Caddy + домен:** `{$DOMAIN}` передаётся в caddy-контейнер через `environment` (иначе SSL не выпустится). `COMPOSE_PROJECT_NAME=tanar-site` в `.env` → детерминированное имя volume (для backup.sh).
- **Перенос каталога:** СИД ОДИН РАЗ. `ALLOW_PROD_SEED=1` (ТОЛЬКО inline, не в .env) ослабляет env-guard; **+ предохранитель в коде: отказ, если каталог не пуст (count>0)** — повторный TRUNCATE на проде физически невозможен. Фото в сид НЕ входят.
- **ИИ-фото:** генерятся ЛОКАЛЬНО → ПАЧКОЙ (3 части: prepare локально → docker cp → сервис `push-media` пишет в БД из контейнера). Фаза deploy-2.
- **Секреты:** `.env*` в `.gitignore` (кроме example). Пароль в `DATABASE_URL` == `POSTGRES_PASSWORD` (два места одного значения — предупреждение в шаблоне).
- **База образа:** `node:20-slim` (Debian), НЕ alpine — sharp под musl/alpine ненадёжен. standalone-артефакт проверяется guard'ом на сборке (`test -f .next/standalone/server.js`). `content/` копируется в runner (MDX блога).
- **Git:** деплой с `main`; merge dev→main — ПЕРВОЕ предусловие шага 7 (под подтверждение).
