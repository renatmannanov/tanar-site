## Задача (исследование + план, НЕ сразу код)

Разобраться с фотографиями товаров и спланировать пайплайн их пополнения через ИИ (nano-banana / Gemini), пригодный и для нас сейчас, и для заказчика через админку потом.

**Сначала исследуй и составь план (через `/write-plan`), не кидайся генерировать.** В конце — согласуй план с пользователем.

### Шаги задачи

1. **Какие фото есть локально.** Источники:
   - `assets/products/<slug>/...` — оригиналы (исходники фотографа / прошлой ИИ-генерации). В gitignore (`/assets/`), лежат локально.
   - `public/images/products/<slug>/<color>/...webp` — обработанные sharp'ом (3 ширины), в git.
   - `src/photo/` — untracked свалка разных jpg/webp (разобраться что это, возможно сырьё).

2. **Соотнеси с товарами на проде.** ВАЖНЫЙ ФАКТ: локальные фото и боевой каталог — **РАЗНЫЕ наборы slug'ов**.
   - Локальные фото-папки (и assets, и public): `hoodie-alatau`, `hoodie-turgen`, `light-jacket-tengri`, `shell-jacket-khan`, `tshirt-tanar` (+ `jacket-sv7-goretex` в public, но БЕЗ подпапок-цветов — 9 файлов навалом, осталось от прошлого).
   - **Реальные товары на проде (12 шт.) — ДРУГИЕ** (см. таблицу ниже). Ни у одного из них на проде НЕТ фото (`media_assets = 0`, витрина на CSS-градиентах).
   - Старые демо-фото были закоммичены в репо и осиротели — их убрали из прод-образа и volume (см. контекст деплоя). Не воскрешать их вслепую — они от других товаров.

3. **Найди пробелы:** для каждого из 12 прод-товаров и каждого его цвета — есть ли подходящий исходник? Скорее всего почти везде НЕТ → надо генерировать. Составь матрицу «товар × цвет × (есть фото? lifestyle? flat-shot на белом?)».

4. **Как добавлять (три случая, через nano-banana):**
   - **4.1 не хватает ЦВЕТА:** взять готовое фото товара и перекрасить в нужный hex (рецепт 2 — перекраска с сохранением геометрии).
   - **4.2 не хватает фото на БЕЛОМ ФОНЕ (flat-shot):** сгенерировать студийный flat-shot из lifestyle-кадра (рецепты 1.a/1.c).
   - **4.3 не хватает ТОВАРА целиком:** придумать схему — напр. запросить обычное фото товара, взять готовое фото модели, «надеть» сгенерированный товар на модель.
   - **Рецепты уже проверены и задокументированы:** `internal/docs/nano-banana-recipes.md` (front/side/back, перекраска по hex — подтверждено 100%). Связанный бэклог: `task_tracker/backlog/flat-shots-pipeline.md` и `assets-pipeline-extension.md`.

5. **Не одноразово, а как инструмент для заказчика.** Шаги 4.1/4.2/4.3 в итоге должны стать функциями **админки** (заказчик сам генерит/перекрашивает фото при создании товара). НО: **сначала отработать вручную/скриптом самим**, убедиться что работает и качество ОК, и только ПОТОМ проектировать интеграцию в админку. План должен это отразить как две фазы: (A) проверка пайплайна руками, (B) интеграция в админку.

### Инструменты (готовы в окружении)
- **nano-banana MCP** сконфигурирован (Gemini API token есть). Тулы: `mcp__nano-banana__generate_image`, `edit_image`, `continue_editing`, `get_last_image_info`. Их схемы подгружаются через ToolSearch.
- **sharp**-пайплайн: `scripts/process-images.mjs` (npm `images`), `scripts/check-images.ts` (npm `images:check`). Обработка → 3×WEBP (1600/800/400).
- Как загрузка в админке кладёт фото: `src/core/media/store.ts` — файл `public/images/products/<slug>/<uuid>-<width>.webp`, строка в `media_assets` (url=1600-вариант, `variantId` = цвет, `role='lifestyle'`, `sortOrder` = append). Чтение для витрины — `@/core/media` (`listProductImages`).
