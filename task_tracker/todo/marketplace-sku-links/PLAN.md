# Ссылки Ozon/Kaspi на каждый размер (marketplace-sku-links)

> Статус: pending
> Дата: 2026-06-11
> Тип: фича

## Цель

У каждого SKU (цвет+размер) — свои ссылки на карточки Ozon и Kaspi (Kaspi не
группирует варианты: одна продуктовая ссылка физически не может быть верной).
Наполняем разовым импортом из таблицы заказчицы, даём редактор в админке,
на витрине кнопки ведут на карточку именно выбранного размера.

## Видение результата

- Покупатель выбрал «Бордо, XL» → кнопки Kaspi/Ozon ведут на карточку этого
  цвета и размера. Размер не выбран → продуктовые ссылки (текущее поведение,
  fallback). У распроданного у нас размера кнопки МП остаются.
- Админка: над формой товара два таба — «Товар» (текущая форма без изменений,
  минус продуктовые поля ссылок) и «Маркетплейсы» (продуктовые fallback-ссылки
  + сводная таблица всех SKU по цветам: Размер / Артикул / Ozon / Kaspi).
  «Сохранить» общий — один submit, один form-state.
- Все 109 SKU заполнены импортом из CSV по артикулу (TANAR-001…109);
  импорт повторяем (idempotent) и работает на проде через tools-сервис.

## Out-of-scope (сознательно НЕ делаем)

- Синхронизация цен/остатков с МП (Фаза 5), WB.
- Валидация, что ссылка живая (HTTP-проверки).
- Отдельный раздел «Маркетплейсы» в сайдбаре админки (отвергнут в обсуждении).
- Автозаполнение продуктовых fallback-ссылок из CSV (продуктовых ссылок в CSV
  нет; поля остаются ручными — на проде сейчас пустые, кнопки до выбора
  размера не показываются, это принято).
- 10 артикулов «уточнить» из подвала CSV — импортируем как есть (решение 2026-06-11).

## Ключевые решения (зафиксированы)

- Хранение: `skus.marketplaces` jsonb `Partial<Record<Marketplace, string>>`,
  NOT NULL DEFAULT '{}' — зеркально `products.marketplaces`. Миграция 0006
  (аддитивная, одна колонка).
- `internal/` в gitignore → CSV не попадает в прод-образ. Путь данных:
  extract-скрипт (локально) → коммитим `src/core/db/marketplace-links.json`
  (только артикул → {ozon, kaspi}, без цен/описаний) → idempotent-сидер
  `seed-marketplace-links` (npm-скрипт + tools-сервис в compose, без
  ALLOW_PROD_SEED — обновляет ТОЛЬКО skus.marketplaces).
- Витрина: `effective = { ...product.marketplaces, ...sku.marketplaces }` для
  выбранного SKU; ключи SKU перекрывают продуктовые, отсутствующие — падают
  на продуктовые. Строка географии («Казахстан — Kaspi») остаётся на
  продуктовых полях.
- Семантика записи: форма — источник правды; `upsertSkus` пишет
  `marketplaces: sku.marketplaces ?? {}` (полная замена, как stockQty).
  Единственные писатели — админ-форма (шлёт реальные значения через mapper)
  и сидер ссылок (точечный UPDATE по артикулу).
- Табы — клиентский useState в ProductForm (НЕ роутинг); неактивный таб
  скрыт через `hidden`-атрибут (НЕ unmount — инпуты controlled, но hidden
  дешевле и не теряет scroll). Submit-футер виден на обоих табах.

## Прод-деплой этого плана

1. `git pull` на сервере, `docker compose ... build --no-cache` (web)
2. **`--profile tools build --no-cache migrate`** — БЕЗ этого migrate бежит по
   старым файлам и врёт «successfully» (грабля 2026-06-10, см. CLAUDE.md)
3. `--profile tools run --rm migrate` (миграция 0006, аддитивная)
4. `up -d`
5. `--profile tools build --no-cache seed-marketplace-links` +
   `--profile tools run --rm seed-marketplace-links` — разовый импорт ссылок
6. Smoke: карточка sv7 → выбрать размер → кнопки Kaspi/Ozon ведут на карточку
   размера
