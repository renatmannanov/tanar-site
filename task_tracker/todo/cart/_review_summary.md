## Критичное (блокирует / даст неверный результат)

### C1. curl-команды верификации в step 4 не работают на Windows/PowerShell
- Найдено в: **code + risks + structure** (тройное совпадение).
- `step_4_order_api.md` строки ~76–83: bash-синтаксис `-o /dev/null`, одинарные кавычки в `-d`. Проект на win32 (PowerShell). На PowerShell `-o /dev/null` создаст файл с именем `/dev/null`, а кавычки сломают JSON.
- Последствие: агент-исполнитель "проверит" шаг с ошибкой, посчитает выполненным, оставит мусорный файл.
- Фикс: переписать на PowerShell (`Invoke-RestMethod` / `curl.exe ... -o $null`) или на `request.post` внутри e2e (как в step 6 сценарий 4 — он уже это делает).

### C2. Неверный порт в curl-командах (3000 vs 3001)
- Найдено в: **code**. Подтверждено чтением `playwright.config.ts`.
- Playwright поднимает dev на **3001** (`npx next dev --port 3001`), а `npm run dev` — на **3000**. Команды в step 4 бьют в 3000.
- Последствие: connection refused если проверять при запущенном тестовом сервере; путаница какой порт верный.
- Фикс: зафиксировать в step 4 что ручная проверка — на 3000 (через `npm run dev`), либо вообще убрать curl и проверять через e2e (3001).

### V3. mounted-паттерн нужен шире, чем только бейдж (риск hydration в catalog.spec)
- Найдено в: **code + risks** (двойное совпадение).
- `catalog.spec.ts` ловит console errors. Если `CartDrawer`/`CartProvider` читают localStorage при первом рендере без guard — hydration mismatch попадёт в этот тест и уронит его.
- Step 3 требует `mounted` только для бейджа CartButton.
- Фикс: в step 1 (CartProvider) и step 3 явно прописать: до маунта рендерим как пустую корзину (items=[]), чтение localStorage только в useEffect. Это закроет hydration целиком, не только бейдж.

### V4. Сервер доверяет цене из тела запроса
- Найдено в: **risks [CONFIRMED]**.
- Step 4 валидирует только `price ≥ 0`, не сверяет с `products.ts`. И `total` пересчитывается из тех же (потенциально поддельных) `price`. Клиент может прислать любую цену.
- Контекст: это витрина без онлайн-оплаты, заказ уходит человеку в мессенджер на ручную обработку → риск низкий. Но дёшево закрыть.
- Фикс (минимальный): в step 4 при сборке Order перечитывать цену по `getProductBySlug(item.slug).price` на сервере, игнорируя клиентскую. Тогда подделка невозможна.

### V5. Нет max-валидации длины тела / нет защиты от спама
- Найдено в: **risks [LIKELY]**.
- `/api/order` публичный, без rate-limit и без max-длины (`contact`, `qty`, `items.length`). Флуд → Telegram 429 / лишние сообщения заказчице.
- Контекст: полноценный rate-limit без БД сложно; для старта достаточно границ.
- Фикс (минимальный, в step 4): добавить max-границы — `contact` ≤ 200 симв, `qty` 1..99 (int), `items.length` ≤ 50. Rate-limit вынести в заметку на будущее, не в этот план.
