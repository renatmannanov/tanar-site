# Шаг 5.2: Генерация привязана к слоту

> Зависит от: шаг 5.1 (сетка 6 слотов, `slotOf`/`PHOTO_SLOTS`)
> Статус: [ ] pending

## Задача

Привязать генерацию к **конкретному пустому слоту** вместо «первого подходящего фото». Это лечит оба косяка чернового UX:
1. **`view` из слота** → back-flat получает правильный промпт (рецепт 1.c с удалением лого), а не front-промпт → нет дорисованного лого на спине.
2. **Источник однозначен** → заказчик контролирует, из чего генерит.

Бэкенд (`@/core/photogen` + 3 экшена `generateFlat`/`recolorFlat`/`recolorLifestyle`) **не переписываем** — он уже принимает `sourceId` и наследует `view`. Меняем: какой `sourceId` и `view` подаём, и где появляются кнопки.

## Логика «заполнить пустой слот»

Кнопка генерации появляется **в пустом слоте** (не общим блоком внизу). Целевой слот = где нажал. Источник = заполненный слот (свой или другого цвета). Маппинг слот → рецепт:

| Пустой целевой слот | Рецепт | Источник (заполненный слот) |
|---------------------|--------|------------------------------|
| `flat_<view>` | **1 (flat)** | `life_<view>` ЭТОГО цвета (тот же ракурс) |
| `flat_<view>` | **2 (recolor-flat)** | `flat_<view>` ДРУГОГО цвета |
| `life_<view>` | **3 (recolor-lifestyle)** | `life_<view>` ДРУГОГО цвета |

- **Ракурс фиксирован слотом:** для `flat_back` источник — только `*_back` (нельзя взять front как back). `view` берётся из целевого слота и передаётся в экшен (а не из поля источника, хотя они должны совпадать).
- **`generateFlat` (рецепт 1)** — единственный, кто меняет роль (life→flat), цвет тот же. `recolor-*` — меняют цвет, роль сохраняют.
- В этом шаге, если у пустого слота **несколько** кандидатов-источников — берём по приоритету: рецепт 1 (свой life) важнее recolor; среди recolor — первый доступный. **Явный выбор источника с миниатюрами — шаг 5.3.** Здесь достаточно детерминированного приоритета (не «первое попавшееся фото», а «свой life того же ракурса, иначе recolor из первого цвета по порядку variants»).

## Что делаем

1. **`view` в экшенах — из целевого слота, не из источника.** Сейчас `media-actions.ts` берёт `view` из `loaded.source.view ?? 'front'`. Изменить: экшены принимают явный `view` в `GenTarget` (целевой ракурс) и передают его в `lifestyleToFlat({view})` / `recolorFlat({view})`. Для recolor-lifestyle `view` тоже из целевого слота (наследуется в upload). Это и есть фикс косяка с лого.
2. **`VariantPhotos`: кнопка(и) генерации в пустом слоте.** Для каждого пустого слота вычислить доступные источники (по таблице выше, используя `assetsBySlot` своего варианта + `siblingImages`). Если источник есть → показать кнопку «заполнить» с понятным текстом («сделать на белом» / «перекрасить из <цвет>»). Источников нет → плитка «+ загрузить» (из 5.1).
3. **Убрать общий блок кнопок внизу** (черновой из шага 5) — генерация теперь в слотах. `genFlat`/`genRecolorFlat`/`genRecolorLifestyle` переиспользовать, но вызывать с `view` целевого слота и конкретным `sourceId`.
4. **Лимит SOFT_MAX=8** на вариант: 6 слотов < 8, лимит не мешает; но если слот занят — кнопки генерации в нём нет (только в пустых). Перезапись занятого слота — шаг 6.

5. **Проверка занятости слота на бэкенде (защита от гонки/двойного клика).** UI прячет кнопку в занятом слоте, но этого мало: двойной клик или гонка может записать второе фото в `(variantId, role, view)` → дубль. Добавить в server actions (`generateFlatAction`/`recolorFlatAction`/`recolorLifestyleAction` **и** `uploadVariantImageAction`) дешёвую проверку ПЕРЕД upload: если в целевом слоте уже есть ассет (`WHERE variantId=… AND role=… AND view=…` вернул строку) → вернуть `{ error: 'Слот уже занят' }`, не генерировать/не грузить. Замена занятого слота идёт ТОЛЬКО через явный flow шага 6 (там удаление старого предшествует записи нового). Хелпер `mediaStore.slotTaken(variantId, role, view): Promise<boolean>` (server-only, в store.ts) — рядом с `get`.
