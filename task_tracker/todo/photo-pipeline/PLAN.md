# Фото-пайплайн каталога (nano-banana / Gemini)

> Статус: in_progress
> Дата создания: 2026-06-04 · Обновлён: 2026-06-05 (5.1–6 уточнены: reorder убран, уникальность слота на бэкенде, base64-превью, ai_generated только на новых)
> Тип: фича (ИИ-генерация фото в админке)

## Цель

Дать заказчику инструмент в админке, который заполняет каталог фотографиями товаров (вместо CSS-градиентов) через генерацию ИИ (Gemini 2.5 Flash Image, он же nano-banana). Заказчик работает мышкой, без участия разработчика и без ручных MCP-вызовов.

## Что выяснили экспериментами (Фаза A — выполнена, см. EXPERIMENTS.md)

Проверили 4 потенциальных сценария на тестовых куртке+худи (`scratch/experiments/`):

| # | Сценарий | Вердикт |
|---|----------|---------|
| 1 | Живое фото → товар на белом фоне (**flat**) | ✅ работает |
| 2 | Flat → перекрасить в другой цвет (**recolor flat**) | ✅ работает |
| 3 | Живое фото → перекрасить товар (**recolor lifestyle**) | ✅ работает |
| 4 | Заменить товар на живом фото (**swap**, надеть др. товар) | ❌ **ОТКЛОНЁН** — переносит криво/ненадёжно |

Рабочие промпты и уроки зафиксированы в `internal/docs/nano-banana-recipes.md` (рецепты 1–3 + правка про удаление тела на side/back).

**Ключевые уроки:**
- side/back генерить из СВОЕГО исходника (не выдумывать из front); явно перечислять части тела к удалению, иначе остаются голова/руки.
- recolor: короткий лобовой промпт + hex, без длинных оговорок.
- swap-донор обязан быть на белом фоне — но даже так результат кривоват → сценарий 4 не делаем.

## Что делаем (Фаза B — основная работа)

Встроить **3 сценария** (1, 2, 3) в админку как кнопки в блоке фото товара (`VariantPhotos`), через прямой вызов Gemini из server action (`@google/genai`). MCP в рантайме недоступен — только Фаза A им пользовалась.

### Архитектура: переиспользуемый движок (решение 2026-06-04)

Генерация выделена в **отдельный модуль `src/core/photogen/`**, не знающий про Tanar (БД, sharp, slug, `media_assets`). Граница — `Buffer/Uint8Array` + параметры (`hex`, `view`, тип рецепта). Цель: переиспользовать движок в других проектах копированием папки.

```
src/core/photogen/          ← ДВИЖОК (reusable, домен-агностик)
  index.ts                    публичный API (generateFlat/recolorFlat/recolorLifestyle)
  provider.ts                 interface ImageGenProvider { editImage(img, prompt): Promise<Buffer> }
  gemini.ts                   class GeminiProvider implements ImageGenProvider (@google/genai)
  recipes.ts                  промпты 1–3 (flat/recolor-flat/recolor-lifestyle), параметризованы view/hex
  types.ts                    GenInput/GenResult, типы view/recipe
  → знает: «фото + параметры → фото». НЕ знает: media_assets, sharp, slug, public/, БД, админку.

src/app/admin/(protected)/catalog/media-actions.ts  ← ИНТЕГРАЦИЯ (Tanar-specific, не переезжает)
  читает media_asset → Buffer → photogen → mediaStore.upload → UPDATE role/view
src/components/admin/VariantPhotos.tsx               ← UI-кнопки (Tanar-specific)
```

Провайдер за интерфейсом `ImageGenProvider` (сейчас одна реализация `GeminiProvider`) — смена ИИ позже = новый файл, рецепты не трогаются. ESLint-границы соблюдены: `photogen` НЕ импортит `core/media`; зависимость только обратная.

### Решения по продукту (согласовано с пользователем 2026-06-04)

- **Пользователь — заказчик** (не разработчик). UX и защита от дурака — приоритет.
- **Режим — свобода + подсказки**, НЕ жёстко зашитый маршрут. Цель: понаблюдать, какие сценарии реально полезны. (Жёсткий маршрут — возможное улучшение потом.)
- **Превью + апрув обязательны** для всех 3 кнопок. Ничего не публикуется автоматически. Это главная защита: «валидность» фото = человеческий глаз, машина её не проверяет.
- **Кнопка доступна только при валидном источнике** (есть нужное фото с нужной меткой `role`/`view`). Нет источника → кнопки нет.
- **Источник выбирается явно**, ракурс (`view`) наследуется от источника автоматически (нельзя выдать перёд за зад).
- **Лимит генераций** — заложить в код, но ВЫКЛЮЧЕН (легко включить позже). На тесте без счётчика.
- **Метка «сгенерировано ИИ»** — и в админке, и на витрине.

### Модель данных (используем существующую, не меняем схему)

Таблица `media_assets` уже имеет нужные поля (`src/core/db/schema.ts`):
- `role` — `'lifestyle' | 'flat'` ← метка типа фото
- `view` — `'front' | 'side' | 'back'` ← ракурс
- `variantId` — привязка к цвету
- `productVariants.hasFlatShots` — флаг «у цвета есть flat»

Проверка «есть ли flat у цвета» = запрос `WHERE variantId=… AND role='flat'`. Надёжно и автоматически.

**Известная недоработка:** ручная загрузка сейчас всегда ставит `role:'lifestyle'`. Нужно дать возможность задать тип при загрузке (или генератор сам проставит правильный role). — закрывается в шаге 5.

## Ключевые контракты (не нарушать)

- **Запись фото — через `mediaStore.upload`** (`@/core/media/store`): sharp в 3 webp + строка `media_assets`. После upload для flat — UPDATE `role/model/view` в ОДНОЙ транзакции (upload ставит `role:'lifestyle'` по умолчанию).
- **Client-компоненты импортят `@/core/media/client`**, не `@/core/media` (barrel тянет sharp → ломает build). `gemini`/`sharp` — только server-only.
- **Генерация — в модуле `@/core/photogen`** (server-only, домен-агностик). Внутри — `GeminiProvider` через `@google/genai`. `GEMINI_API_KEY` уже в окружении (env var), но НЕТ в `.env.example`/`.env.prod.example` — добавить. `photogen` НЕ зависит от `core/media` (только обратная зависимость).
- **Прод-заливка фото — отдельный план** (prod-deploy step 9, `scripts/push-media.ts`). Этот план фото на прод НЕ заливает.
