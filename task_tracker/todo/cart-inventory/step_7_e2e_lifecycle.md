# Шаг 7: Сквозной e2e — жизненный цикл остатка через заказ

> Зависит от: шаги 2, 3, 4, 5 (вся механика и UI готовы)
> Статус: [ ] pending

## Задача

Новый спек `e2e/inventory-lifecycle.spec.ts` (serial; логин/создание товара —
паттерны из cart.spec/admin-orders.spec). Самодостаточный товар:
«Тестовый Цикл X3» (slug `testovyy-tsikl-x3`), 1 цвет, размер M, **stockQty = 2**,
published.

Сценарий (один serial-поток, наблюдаемость — ТОЛЬКО через UI):

1. **Резерв сушит витрину.** Витрина: M → корзина, qty 2 («+» в drawer) → оформить
   (№1). Админка: №1 → «Подтверждён» (ok). Витрина (reload карточки): M
   перечёркнут, `ask-restock` виден (available 2−2=0).
2. **Распродано на витрине.** Витрина: клик по перечёркнутому размеру M
   ПОКАЗЫВАЕТ `ask-restock` вместо `add-to-cart` (smoke-ассерт — детально это
   проверено шагом 4). Очистить localStorage страницы (page.evaluate) → корзина пуста.
3. **Отмена возвращает.** Админка: №1 → «Отменён» (ok). Витрина (reload): M снова
   доступен, `stock-indicator` с `data-level="low"` (available 2).
4. **Повторный резерв и списание.** Витрина: M qty 1 → оформить (№2). Админка:
   «Подтверждён» → «Выполнен» (оба ok). Витрина: M доступен, `data-level="low"`
   (available 2−1=1 — после done списание: stock 1, reserved 0).
5. **Симметрия переходов.** (same→same через селект не выбрать — идемпотентность
   остаётся конструкцией кода: ранний return `old === new` под локом, e2e её
   не проверяет.) Админка: №2 «Отменён» (ok) → витрина (reload): M доступен,
   `stock-indicator` виден. Админка: №2 снова «Выполнен» (ok) → витрина (reload):
   M доступен; добавить M в корзину → в drawer «+» disabled на qty 1, виден
   `qty-limit` (= available ровно 1, как после п.4 — туда-обратно остаток
   не дрейфует). Очистить localStorage.
6. **Гигиена удаления.** Админка: удалить заказ №2 крестиком — статус done:
   confirm-диалог содержит «Списанный остаток на склад не вернётся» (ассертить),
   остатки НЕ возвращаются (витрина после reload: M по-прежнему available 1,
   `data-level="low"` — списание сохранилось). Удалить заказ №1 (cancelled —
   без движений). afterAll: удалить товар (заказы уже удалены — FK не мешает).
7. **Удаление подтверждённого снимает резерв.** Витрина: M qty 1 → оформить (№3).
   Админка: №3 → «Подтверждён» → витрина: M распродан (available 0). Удалить №3
   крестиком → витрина: M снова available 1. afterAll см. п.6.

SQL-ассертов в спеке нет — вся проверка через витрину/админку. (Ручная
SQL-сверка inventory_log — критерий PLAN.md, выполняется при завершении.)
