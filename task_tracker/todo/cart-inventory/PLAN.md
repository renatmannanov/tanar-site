# Корзина ↔ остатки (cart-inventory, Фаза 2 lite)

> Статус: pending
> Дата: 2026-06-11
> Тип: фича

## Цель

Связать витрину и заказы с уже заведёнными остатками (`skus.stockQty`/`reservedQty`,
96/109 SKU со стоком — и в dev, и на проде): покупатель видит доступность и не может
заказать распроданное, заказчица двигает остатки сменой статуса заказа, все движения
журналируются в `inventory_log`.

## Видение результата

- Карточка товара: выбран цвет+размер → слева внизу «В наличии •» с точкой-светофором
  (зелёная ≥10, оранжевая 3–9, красная 1–2 доступных); цифры покупателю НЕ показываются.
  Строки «Алматы — заказ через корзину…» и «Возврат 30 дней.» выровнены вправо.
- Распроданный размер (доступно 0): кнопка размера перечёркнута, но кликабельна;
  при выборе вместо «В корзину» — wa.me-ссылка «Узнать о поступлении» (с товаром,
  цветом и размером в тексте).
- Drawer: «+» упирается в доступный остаток, под степпером — «Больше нет в наличии».
- `createOrder` отказывает при нехватке (`stockQty - reservedQty < qty`) —
  существующая UI-ветка `unavailableSkuIds` подсвечивает позиции.
- Админка: «Подтверждён» → резерв; «Выполнен» → списание; «Отменён» → сброс резерва;
  обратные переходы симметричны (модель эффектов, см. step_1). Подтверждение при
  нехватке блокируется — селект откатывается, рядом текст «Не хватает остатка: …».
  Удаление подтверждённого заказа снимает резерв; confirm-диалог удаления
  предупреждает по статусу: confirmed — «Резерв будет снят», done — «Списанный
  остаток на склад не вернётся».
- Ручная правка остатка в форме товара пишет `inventory_log` (reason=manual).
- Все движения транзакционны (row-lock заказа + SKU) и идемпотентны
  (эффект вычисляется из старого статуса внутри транзакции).

## Out-of-scope (сознательно НЕ делаем)

- Раздел «Остатки» в админке — правка уже есть в форме товара (колонка у SKU).
- Резерв при оформлении или при добавлении в корзину; TTL-резервы. Резерв — ТОЛЬКО
  на подтверждении админом (решение зафиксировано 2026-06-11).
- Показ остатка цифрами покупателю.
- Просмотр inventory_log в админке (журнал пишем, UI — бэклог).
- Синхронизация остатков с маркетплейсами (Фаза 5).
- Миграции БД — НЕ нужны: колонки и таблица inventory_log существуют с миграции 0000.

## Ключевые решения (зафиксированы)

- Доступно = `stockQty - reservedQty`.
- Пороги точек (константы в `@/core/inventory/client`): зелёный `available >= 10`,
  оранжевый `3..9`, красный `1..2`, `<= 0` — распродан. (Заказчик: «больше 10 /
  9–3 / меньше 3»; значение 10 отнесено к зелёному.)
- Модель эффектов статуса: `pending`/`cancelled` → нет эффекта; `confirmed` →
  resервировано; `done` → списано. Переход = снять эффект старого статуса +
  применить эффект нового. Покрывает ЛЮБОЙ прыжок селекта (включая done→pending).
- `inventory_log.delta` — штуки со знаком; для reason `sale`/`return`/`manual` —
  движение `stock_qty`, для `reservation`/`reservation_release` — движение
  `reserved_qty` (+ = резерв вырос). Семантика фиксируется комментарием в schema.ts.
- Проверка доступности — при наборе резерва и при списании без резерва; переходы,
  не уменьшающие available (release, return), не проверяются и не блокируются.
- `CartItem.available` — снапшот доступного на момент добавления (опциональное поле,
  старые корзины без поля живут без клиентского лимита — сервер всё равно проверит).
